## 第0章 ドキュメントの位置づけと前提

### 0.1 本書の位置づけ

本書は、ゼロから新規に構築する自動開発ツール **AutoDev for Android** の **要件定義書（最終確定版）** である。何を作るか（What）、なぜ作るか（Why）、どのような制約のもとで作るか（Constraints）、どのような受け入れ基準を満たさなければならないか（Acceptance）、および詳細な実装方針（How）を1ファイルに集約する。本書は単独で読めば仕様を把握できる自己完結文書として作成されており、過去版（v1.0〜v4.1）の参照を必要としない。

最終的な実装の正は本ツール自身の dev 環境上の実装である。本書は着工時の上流仕様だが、実装が固まった後は本書も実装に同期される下流成果物となる。要件不備が露見した場合は、本書を凍結された契約として扱わず、ループの通常入力として更新する。

### 0.2 用語の暫定定義（詳細は第17章）

- **AIエージェント**: 本書では `claude --dangerously-skip-permissions` コマンドで起動された Claude のセッションを指す。
- **AutoDev for Android**: 本書で定義する自動開発ツール全体の呼称（仮称）。本体は `autodev/` 配下および `.claude/` 配下に集約される。
- **イテレーション**: 1タスク分の最小開発単位。コミットで閉じる。
- **機能ループ**: 1機能の開発を完結させるループ。複数イテレーションと最終実機検証を含み、コミット・プッシュで閉じる。
- **修正ループ**: 1件の修正対応（要件変更・バグ修正）を完結させるループ。
- **システムループ**: システム全体開発フロー（S1〜S15）の総称。
- **dev 環境**: 開発・実機検証を行う実環境。本ツールでは原則として開発マシン上の単一のローカル環境を指す。
- **実機検証**: 実機（エミュレータ/実機）で画面を起動し動作確認すること、実 HTTP で API を叩くこと、本番初期化経路で構築した実 DB に対して動作確認すること。Robolectric / プレビュー / mockでの確認は実機検証に含まれない。
- **本番初期化経路**: `init_db.py` のように、本番運用と同じ手順で DB を構築する経路。
- **ペルソナ駆動テスト**: 事前定義された複数ペルソナを基に、AIが各ペルソナの視点で実機シナリオを実行し、改善要望を列挙して次フェーズの修正タスク列に変換するテスト形態。
- **開発進捗状況.md**: 状態ファイル群（autodev/state/ 配下）の最新内容を人間可読サマリとして集約した Markdown ファイル。`autodev/開発進捗状況.md`。状態 JSON の更新と同期して随時再生成される。
- **Android 開発環境**: setup.sh が自動導入する JDK 17 / Android SDK / AVD / Gradle wrapper / Maestro の総称。版数は `autodev/config/android_env.yaml` にピン留め。
- **Maestro / Compose UI Test**: 実機（エミュレータ/実機）による E2E 検証の標準スタック。
- **ラッパースクリプト**: AIエージェントが素のコマンド（python、pip、JUnit、maestro 等）を直接呼ばないよう、必ず経由するシェルスクリプト群。`autodev/scripts/run_*.sh`。
- **hooks**: Claude Code のライフサイクルイベント（PreToolUse / PostToolUse / SessionStart / UserPromptSubmit 等）で自動実行されるスクリプト群。`.claude/settings.json` で宣言する。
- **skills**: Claude Code が特定タスクで自動発動する手順書ディレクトリ。`.claude/skills/<skill名>/SKILL.md` の形で配置する。
- **attestation**: 完了の証拠付き宣言。誰が何をどの実機検証で確認したかを記述し、証跡へのパスを含む。
- **staleness ゲート**: 設計書と実装の乖離を機械検出する仕組み。
- **scoped tests**: 変更領域 + 影響範囲 + 全体スモークに限定したテスト。
- **DRI**: 説明責任者（Directly Responsible Individual）。ハーネスの整合に責任を持つ1人。
- **ground-truth 再実行**: 自己申告を信用せず、ハーネス側で同じテストを再実行して実態を確認する仕組み。
- **全体設計書**: システムループ S4 で AI が新規作成するシステム全体の機能間関係・データフロー・共通機能設計書。`docs/設計/全体設計書.md`。
- **design_doc_ref**: タスクが参照する設計書のセクション（ファイルパス + 見出し or 行範囲）。タスク分割漏れ防止のため必須。
- **顧客提供物**: リリース時にユーザーから顧客に引き渡される成果物。app/、maestro/、gradle一式、docs/設計/、README.md、操作手順書、デプロイ手順を含む。`autodev/` 配下と `.claude/` 配下は含まれない。

## 第1章 背景と目的

### 1.1 背景

既存の自動開発ツール群（automated_inspection・project_template とその派生）を運用した結果、以下の事故クラスが繰り返し観測された。

1. DB スキーマがマイグレーション定義／自前 CREATE／repo の3系統に分裂し、整合主体不在で並存した（Web 版での教訓）。
2. 環境不在時に別環境へ silent fallback し、依存欠落のままテストが PASS／skip して「緑」になった（Web 版での教訓）。
3. 共有資源（ビルドキャッシュ / 生成物 `build/`）が symlink で worktree 間に伝播し、main ブランチに混入した（Web 版での教訓）。
4. フロントエンドの「緑」が Robolectric + Robolectric/JVMテスト 代替に依存しており、実機（エミュレータ/実機）での画面確認（Maestro / Compose UI Test スクショ・visual baseline）が事実上ゼロだった。
5. 並列実行に起因する dangling commit、データ消失、false BLOCKED が多発した。
6. worker / reviewer の自己申告（「167 passed」等）が実態と乖離しても気付かれなかった。
7. 設計書（上流の正）と実装が乖離しても乖離を検知する仕組みが無かった。
8. 旧モデル時代の回避策が陳腐化したまま棚卸しされず、対症療法を恒久策と誤認して再発を助長した。
9. 自動開発ツール本体のプログラムが開発対象アプリのコードベースに混在し、リリース時の分離が難しかった。
10. タスク分割時に設計書との対応が不明確で、設計書に書かれたが実装されない（または逆）要素が生じた。
11. 機能リストが要件定義書から素直に拾えるものに限られ、基盤・共通機能の欠落がフェーズ後半で露見した。
12. ペルソナの視点でのユーザー検証が形式化されておらず、開発側視点だけで「動くもの」を作って終わっていた。
13. 開発進捗が状態ファイル（JSON 等）に閉じ、人間がリアルタイムに把握する手段がなかった。
14. Maestro / Compose UI Test のセットアップが個別ループで都度行われ、AIエージェントが mock や Robolectric にすり替えて「緑」を作りやすかった。
15. Python の仮想環境を作らずに system Python で開発が進み、依存欠落・ライブラリ衝突が頻発した。
16. ハーネス規約が CLAUDE 規約のテキストにのみ存在し、AIエージェントが規約遵守を「読んで覚える」前提だったため、長時間運用でドリフトが起きた。
17. 各ループで類似手順を AI が毎回コンテキストから再構成しており、手順が微妙にぶれた。
18. 長時間の自動開発でセッションのトークン消費が肥大し、コンテキスト溢れで AI が直近のタスクしか覚えていない状態（および費用増加）が発生した。

これらは知見メモの観測事案として整理されており、いずれも「上流凍結ゲート方式」「並列開発」「設計書を上流の正にする運用」「self-report を信用する運用」「ハーネス固定資産化」「ハーネスのテキスト依存」の組合せから生じる構造的事故であった。

### 1.2 目的

上記事故クラスを構造的に発生不能にし、かつ既存ツールにありがちな「重く・遅く・止まる」運用を避けて、**「素早く・小さく・実機で確かめながら・ペルソナの視点でも検証しながら・進捗を人間に見える形で・規約遵守を仕組みで強制し・節目で会話圧縮しながら」** 開発を進める自動開発ツールを、既存ツールとは独立にゼロから構築する。さらに、本ツール自体の成果物が顧客リリース時に容易に分離・削除できる構造を取る。

### 1.3 ゴール

本ツールが達成すべきゴールを以下に定める。

1. ユーザーが事前に **要件定義書.md** / **機能要件_(機能名).md** / **修正_(修正名).md** のいずれかを所定ディレクトリに配置し、AIエージェントに対して **たった1回のプロンプト** を投げるだけで、対象システム／機能／修正の対応が完了するまで全自動で進む。
2. 開発途中でユーザーに確認・承認・人手介入を求めない（ただし対話可能性は失わない。第6章 6.2.1 参照）。
3. **機能単位** で開発し、機能の動作が確認できた時点で **実機（エミュレータ/実機）による実機動作確認** を必ず実施する。
4. テストは **変更領域に限定** し、網羅テストは別フェーズ扱いとして時間を節約する。ただし実機検証の必須性は不可侵。
5. UI は **Jetpack Compose**、言語は **Kotlin**、DB は **Room（端末内 SQLite）**、設定は **DataStore** で実装される。
6. AIエージェントは **`claude --dangerously-skip-permissions`** で起動されるものに限定し、`claude -p` のような追加料金が発生する起動形態は使わない。
7. **本ツール本体（プログラム・スクリプト・ハーネス・状態ファイル・ユーザー入力）はすべて `autodev/` 配下と `.claude/` 配下に集約**する。リリース時にはこれら2つのディレクトリを削除するだけで顧客提供物から本ツール関連を一括除外できる。
8. **ペルソナ駆動テストを正式フェーズとして組み込む**: 全体設計時に複数ペルソナを定義し、3機能ごと + 全機能完了後にペルソナ視点での実機テスト・改善要望反映を行う。
9. **全体設計書を機能ループ前に作成**し、各機能のタスクは設計書のセクションと紐付ける。設計書に書かれた要素が漏れなくタスク化される構造を取る。
10. **開発進捗をリアルタイムで人間可読化**する: 状態ファイル更新時に `autodev/開発進捗状況.md` を同期更新する。
11. **Python仮想環境の徹底**: バックエンドのすべての Python / pip 実行は `JDK/Android SDK（setup.sh が自動導入）` 経由とし、システム Python へのフォールバックを構造的に禁止する。
12. **Maestro / Compose UI Test の常時セットアップ完了状態**: 初期状態の時点で実機（エミュレータ/実機）テストが実行できる状態にし、AIエージェントが実機確認に必ず Maestro / Compose UI Test を使用するよう運用を徹底する。
13. **Claude Code の hooks を活用**: ハーネス遵守（ラッパー経由運用、進捗.md 同期、各種ゲート、リリース前クリーン）を AIエージェントの「読んで覚える」運用に頼らず、PreToolUse / PostToolUse / Stop / SessionStart / PreCompact 等の hooks で自動強制する。
14. **Claude Code の skills を活用**: システムループ / 機能ループ / 修正ループ / ペルソナ駆動テスト / Gradle・ADB 運用（autodev-gradle-env）/ 実機検証（autodev-device-verify）などの標準手順を skills（SKILL.md）として独立配置し、AIエージェントが該当タスクに入った時に自動発動・適用する。
15. **会話圧縮は Claude Code 組み込みの自動 compact に委ねる**: `/compact` はユーザー入力専用で自動化できないため、ツールからは撤去済み。コンテキスト逼迫時は組み込みの自動 compact が常時作動し、長時間自動開発でも安定する。

### 1.4 非ゴール

以下は本ツールの非ゴールである。

- 既存ツール（automated_inspection・project_template）との互換性維持／移行支援。
- 複数AIエージェントによる並列開発／オーケストレーション。
- 多人数・多リポジトリへの設定配布／組織横断ガバナンス。
- Word/PPT 等のドキュメント自動生成（本書が定めるのは Web アプリ開発ツールであり、ドキュメント生成ツールではない）。
- AIエージェント自身の学習・モデルチューニング。
- 顧客向けの本番デプロイ自動化（dev 環境で動くまでを保証する）。
- Claude Code 以外の AI エージェント基盤（Cursor / Aider / Cline 等）への移植。本ツールは Claude Code の hooks / skills 機能に依存する。

### 1.5 中核思想

以下の14原則を本ツールの中核思想として全章に貫徹する。

1. **開発環境上の実装こそ唯一のマスタ**: 設計書・テスト期待値・受け入れ判定は、すべて「動いている dev 環境の実装」から導出・検証される。上流を凍結しない。
2. **直列開発のみ**: 並列は事故の増幅器。常設の単一 dev 環境を1本のループで進める。
3. **機能の開発ごとに実機検証を必須として含める**: 1機能の開発は「実機（ブラウザ上での画面表示確認など）で動作確認できて初めて閉じる」。Robolectric / mock / デッドコードの緑で閉じることを構造的に禁止する。
4. **設計書は実装に同期させる下流成果物**: ただし全体設計書および機能設計書の初版は機能開発前に新規作成可能（実装が成立した時点で実装が正となる）。
5. **要件変更を一級市民として扱う**: ループが吸収する。凍結ゲートで阻まない。修正対応も同様に正式なループ入力とする。
6. **ツール本体と顧客提供物の物理的分離**: ツール関連はすべて `autodev/` 配下と `.claude/` 配下に集約。コマンド2つで分離可能。
7. **ペルソナ視点の必須化**: 開発側視点だけでない、ペルソナの視点での実機検証を正式フェーズに組み込む。
8. **タスクと設計書の双方向紐付け**: タスク分割は設計書のセクションと1対1以上で紐付け、分割漏れ・実装漏れを構造的に防ぐ。
9. **開発進捗の人間可読化（リアルタイム）**: 状態ファイル群はAIエージェントとハーネスの間の機械可読契約だが、それだけでは人間（開発者）が状況把握できない。状態の更新と同期して `autodev/開発進捗状況.md` を生成・更新する。
10. **ビルド・端末操作は必ずラッパー経由**: 環境迂回・silent fallback は silent-pass 事故の温床。gradle/adb/maestro のラッパー経由実行をラッパースクリプト + ヘルスチェック + 規約 + hook の多層で強制する。
11. **Maestro / Compose UI Testは初期状態でセットアップ済み**: 各ループで「セットアップが面倒だから今回は mock で代用」と判断する余地を構造的に消す。初期雛形時点で `maestro install` 済み、初回 spec 通過済みの状態を必須とする。
12. **ハーネス遵守は hooks で機械強制する**: 「規約に書いてあるから AI は守る」という運用前提を捨てる。PreToolUse hook で禁止コマンドを fail させ、PostToolUse hook で同期処理を自動実行する。
13. **標準手順は skills で独立配置する**: 各ループ・各検証の手順は SKILL.md として独立ファイル化し、AIエージェントは該当タスクに入った時にその skill を自動発動・適用する。手順のドリフトを防ぐ。
14. **会話圧縮は組み込み自動 compact に委ねる**: `/compact` はユーザー入力専用で自動化できないため、ツールからは扱わない。コンテキスト逼迫時は Claude Code 組み込みの自動 compact が常時作動し、長時間運用でも安定して進める。

## 第2章 システム概要

### 2.1 AutoDev for Android の構成要素

AutoDev for Android は以下の構成要素から成る統合ツールである。

**本ツール本体**（`autodev/` 配下 + `.claude/` 配下、リリース時に両方削除可能）:

1. **ループ制御ハーネス**: CLAUDE 規約、状態ファイル、ゲートスクリプト。
2. **入力規約と入力ファイル**: 要件定義書 / 機能要件書 / 修正対応書のフォーマット規定とユーザー作成ファイル。
3. **検証スタックの設定**: Maestro / Compose UI Test 設定、JUnit 設定、scoped tests、ヘルスチェック。
4. **証跡保存域**: スクショ・HTTP ログ・DB ダンプ・Maestro / Compose UI Test トレース。
5. **状態管理域**: ループ状態・タスク状態・attestation・ペルソナ・進捗履歴。
6. **テンプレート**: 入力ファイル雛形、初期アプリ雛形。
7. **進捗ダッシュボード**: `autodev/開発進捗状況.md`（状態 JSON 更新と同期）。
8. **共通ラッパースクリプト群**: `run_*.sh` / `check_*.sh` / `setup_*.sh`。
9. **Claude Code hooks**: `.claude/settings.json` で宣言される PreToolUse / PostToolUse / Stop / SessionStart / PreCompact 等のライフサイクルフック。実体スクリプトは `autodev/hooks/claude/` 配下に集約。
10. **Claude Code skills**: `.claude/skills/<skill名>/SKILL.md`。実体は `autodev/skills/<skill名>/` 配下に集約し、`.claude/skills/` からはシンボリックリンクまたはビルド時コピーで参照する。

**操作対象**（リポジトリ直下、顧客提供物の候補）:

11. **アプリ実装**: `app/`（Kotlin + Jetpack Compose + Room + DataStore の単一モジュール。ui / data / domain / settings / util 構成）。
12. **設計書**: `docs/設計/`（実装の地図として実装から同期される下流成果物）。
13. **顧客向けドキュメント**: `README.md`、`docs/操作手順書.md`、`docs/デプロイ手順.md` 等。

なお、AIエージェント本体（Claude）は外部実行環境であり本ツールの構成要素ではない。

### 2.2 リリース時の分離原則

ルート直下の **`autodev/` ディレクトリと `.claude/` ディレクトリの2つを削除するだけで、自動開発ツール関連が全て消える**。

```
rm -rf autodev/                # ハーネス本体・入力・状態・証跡・ログ・進捗.md・hooks/skills 実体
rm -rf .claude/                # Claude Code 設定・hooks 宣言・skills 配置
rm -f CLAUDE.md                # ルートの最小ポインタ
# 残った app/ maestro/ docs/設計/ README.md などが顧客提供物
```

詳細手順は付録I。

### 2.3 利用イメージ（最短）

新規システム開発:
```
$ cd <project-root>
$ vi autodev/inputs/要件定義/要件定義書.md
$ claude --dangerously-skip-permissions
> 要件定義書.mdのシステムを自動で開発してください
（以降、Claude Code の hooks / skills が AIエージェントの動作を導きつつ全自動進行）

# 別ターミナル/別端末で:
$ tail -f autodev/開発進捗状況.md       # リアルタイム進捗確認
```

機能単位:
```
$ vi autodev/inputs/要件定義/機能要件_ログイン.md
$ claude --dangerously-skip-permissions
> 機能要件_ログイン.mdの機能を自動で開発してください
```

修正対応:
```
$ vi autodev/inputs/修正対応/修正_ログインボタン無反応.md
$ claude --dangerously-skip-permissions
> 修正_ログインボタン無反応.mdの修正を自動で対応してください
```

## 第3章 スコープ

### 3.1 対象範囲（in scope）

- Android アプリ（Kotlin + Jetpack Compose + Room + DataStore）の **新規開発**・**機能追加**・**要件変更/バグ修正対応** を、AIエージェント単独で全自動進行させる仕組み。
- 開発ループ、修正ループ、実機検証ループ、ペルソナ駆動テスト、全体設計書作成、設計書同期、状態管理、ログ・証跡、ハーネスの自動強制（hook / ゲート）。
- 入力フォーマット（要件定義書.md、機能要件_*.md、修正_*.md）の規約と、それらを起点とする起動プロトコル。
- DB スキーマ管理と、SQLite ⇄ MySQL の切替容易性を担保する抽象層。
- 単一 dev 環境上での実行・実機検証（Maestro / Compose UI Test）。
- 本ツール本体の `autodev/` + `.claude/` 集約による顧客提供物との物理的分離。
- システム完成時の顧客向けドキュメント（README、操作手順書、デプロイ手順）の自動整備。
- 開発進捗の人間可読化（autodev/開発進捗状況.md の自動生成・同期）。
- Python仮想環境を必須前提とした自動セットアップとラッパー運用。
- Maestro / Compose UI Test のセットアップ完了状態の保証と運用徹底。
- Claude Code の hooks / skills 機能を活用したハーネス強制と標準手順配信。
- 会話履歴圧縮は Claude Code 組み込みの自動 compact に委ねる（`/compact` はユーザー入力専用のためツールでは扱わない）。

### 3.2 対象外（out of scope）

- 本番リリース・デプロイ自動化（CI/CD、ステージング、本番昇格）。本ツールは「dev 環境で動くまで」を保証する。
- マルチユーザー協調開発、コードレビュー協調、PR ワークフロー。
- モバイルネイティブアプリ、デスクトップネイティブアプリ。
- 機械学習モデルの学習・サービング。
- 監視・運用ツール（APM、ログ集約 SaaS 等）との統合。
- AIエージェントの API キー管理・課金管理（ユーザーの Claude 環境に依存）。
- 顧客リリース時の難読化・パッケージ署名・配布管理（分離だけを担保し、配布は対象外）。
- Claude Code 以外の AI エージェント基盤への移植。

### 3.3 想定アプリ規模

- 1機能 ＝ 数〜十数イテレーション。
- 1イテレーション ＝ 数十分〜数時間の AI 稼働。
- 1システム ＝ 数機能〜十数機能。
- 1修正対応 ＝ 1〜数イテレーション。
- 1ペルソナ駆動テスト ＝ 数ペルソナ × 数シナリオ。
- 単一リポジトリ。マルチリポは非対応。

## 第4章 利用者像と利用シナリオ

### 4.1 ペルソナ

#### 4.1.1 一次利用者: 個人開発者 / 小規模チームの技術者

- 本ツールの存在を意識しないこと自体が要件。

### 4.2 利用シナリオ

#### シナリオA: 新規システムをゼロから開発する

1. ユーザーが本ツールのプロジェクト雛形を clone（または相当のコマンドで生成）。雛形には `autodev/` と `.claude/` 配下が完成形で同梱されている。
2. ユーザーが `autodev/inputs/要件定義/要件定義書.md` を編集し、システム全体の要件を記述する。

7. 顧客リリース時には `rm -rf autodev/ .claude/` でツール本体を削除し、残った成果物を顧客に提供する。

#### シナリオB: 既存システムに機能を追加する

1. ユーザーが `autodev/inputs/要件定義/機能要件_<機能名>.md` を作成。
2. ユーザーが `claude --dangerously-skip-permissions` を起動して「機能要件_<機能名>.mdの機能を自動で開発してください」とプロンプト送信。
3. AIエージェントが機能ループを1回回す。F2 で当該機能の設計書が無ければ新規作成、F3 でタスクと設計書セクションを紐付けてから F4〜F6 を実行、**F6.5 で新機能を主動線としたペルソナ駆動テストを実施し、改善要望（機能改善 / UI 改善）を自動抽出してサブ修正ループで消化**（FR-5.9.8）。F7 でクローズ（ゲート→attestation→コミット・プッシュ）。

#### シナリオC: 修正対応（要件変更・バグ修正）

1. ユーザーが `autodev/inputs/修正対応/修正_<修正名>.md` を作成。
2. ユーザーが「修正_<修正名>.mdの修正を自動で対応してください」とプロンプト送信。
3. AIエージェントが修正ループ R1〜R7 を実行: 影響範囲特定 → 関連設計書確認 → タスク分割と設計書紐付け → イテレーション実行 → 実機検証 → **R6.5 で影響範囲動線中心のペルソナ駆動テストを実施し、改善要望（機能改善 / UI 改善 / リグレッション）を自動抽出してサブ修正ループで消化**（FR-5.9.9）→ 設計書同期 → コミット・プッシュ。
4. R6.5 のペルソナテスト・改善対応がすべて完了した上で、修正ループ完了時点で停止。

#### シナリオD: 要件不備が実機検証で露見した

1. AIエージェントが機能ループの実機検証段階で要件不備を検出。
2. 機能要件書を読み直し、要件不備を AI 自身がループ入力に変換する（要件補足追記）。
3. 補足された要件を反映した実装を再度ループに通す。
4. ループのクローズ条件（実機検証 PASS）を満たしたら閉じる。

#### シナリオE: ペルソナ駆動テストで改善要望が出た

1. システム全体開発フローの S7 / S9 で、3機能完了後にペルソナ駆動テストフェーズが起動。
2. AIエージェントが事前定義した複数ペルソナを順次ロールプレイし、典型シナリオを実機（エミュレータ/実機）で実行。
3. 各ペルソナ視点で改善要望を列挙し、修正ループ用の入力ファイル `autodev/inputs/修正対応/修正_<改善名>.md` として自動生成。
4. 各改善要望に対して修正ループを順次実行。
5. 全改善要望が対応完了したら次の3機能の開発フェーズへ進む。

#### シナリオF: 進捗を別端末で随時確認する

1. ユーザーが PC で `claude --dangerously-skip-permissions` を起動し、自動開発をスタート。
2. ユーザーが iPad / 別 PC などで `autodev/開発進捗状況.md`（クラウド同期もしくは git pull）を Markdown プレビューで開く。

4. 異常を検知した場合のみ、ユーザーは PC に戻って AI に質問・指示を出す。

1. ユーザーが起動プロンプトを送信し、システム全体開発（機能数 10〜20）が走り出す。

4. ユーザーは `autodev/開発進捗状況.md` を見るだけで状況把握。

## 第5章 機能要件

各 FR は ID（FR-x.y.z）を持ち、第14章の受け入れ条件と紐づく。

### 5.1 起動・入力

- **FR-5.1.1 起動コマンド**: 本ツールは `claude --dangerously-skip-permissions` で起動されたAIエージェントセッション上で動作する。それ以外の起動形態（`claude -p` 等）は使用してはならない。
- **FR-5.1.2 起動時 CLAUDE 規約読込**: プロジェクトルートに配置された `CLAUDE.md`（最小ポインタ）が、AIエージェント起動時に自動読み込みされ、そこから `autodev/CLAUDE_MAIN.md` および各サブ規約に追加ロードされる。
- **FR-5.1.3 起動プロンプト3形態**: ユーザーは以下のいずれか1つのプロンプトを起動直後に1回だけ送信することで自動開発を起動できる。
  - 「**要件定義書.mdのシステムを自動で開発してください**」: `autodev/inputs/要件定義/要件定義書.md` を入力とするシステム全体開発モード（フロー: 9.2）。
  - 「**機能要件_(機能名).mdの機能を自動で開発してください**」: `autodev/inputs/要件定義/機能要件_<機能名>.md` を入力とする機能単位開発モード（フロー: 9.3）。
  - 「**修正_(修正名).mdの修正を自動で対応してください**」: `autodev/inputs/修正対応/修正_<修正名>.md` を入力とする修正対応モード（フロー: 9.4）。
- **FR-5.1.4 起動プロンプトの揺れに対する耐性**: 上記3形態に意味的に等価な日本語表現も受理可能。判定の曖昧さがある場合は質問する。
- **FR-5.1.5 入力ファイルの自動探索**: 起動プロンプトで明示されたファイル名（または機能名・修正名）に対応するファイルを `autodev/inputs/` 配下から AI が探索して見つける。複数候補・該当なしの場合は質問する。
- **FR-5.1.6 入力ファイルテンプレート**: `autodev/inputs/_テンプレート/` 配下に3形態のテンプレートを同梱する（付録A）。
- **FR-5.1.7 入力ファイル形式チェック**: 起動時に入力ファイルの必須セクション（11章参照）が揃っているか機械チェックし、欠落があれば質問または明示的に注釈を付けて続行する。

### 5.2 自動開発ループ

- **FR-5.2.1 ループの階層構造**: 本ツールのループは以下の階層構造をとる。詳細は第9章。
  - **システムループ**: 要件定義書を入力に、ペルソナ定義から完了報告までの S1〜S15 フローを実行する。
  - **機能ループ（F1〜F7）**: 1機能の開発を完結させる。複数イテレーション + 機能全体レビュー + 機能実機検証 + プッシュ。
  - **修正ループ（R1〜R7）**: 1件の修正対応を完結させる。機能ループに準じる構造を持ち、影響範囲分析を強化する。
  - **イテレーションループ（I0〜I8）**: 1タスクの小さな変更 + 設計書同期 + 変更箇所レビュー + 変更箇所テスト + コミット。
- **FR-5.2.2 直列実行**: ループは常に直列で実行される。複数機能・複数イテレーション・複数修正を並列化してはならない。
- **FR-5.2.3 タスク分割と設計書紐付け**: 機能ループおよび修正ループの開始時、AIエージェントは要件を1イテレーションで実装可能な粒度のタスク列に分割する。**各タスクには対応する設計書のセクション参照（ファイルパス + 見出しレベル or 行範囲）を `design_doc_ref` として必ず持たせる**。タスクと設計書セクションの対応関係を `autodev/state/tasks/<対象名>.json` に記録し、設計書の全セクションが少なくとも1つのタスクから参照されていることを `check_task_design_link.sh` で機械チェックする（分割漏れ防止）。
- **FR-5.2.4 タスク進捗管理**: 各タスクは pending → in_progress → done のステータスを持ち、`autodev/state/tasks/<対象名>.json` で atomic に更新される。
- **FR-5.2.5 ループのクローズ条件**:
  - イテレーションループ: 変更箇所のコード・設計書レビュー PASS、変更箇所のテスト PASS、ground-truth 再実行 PASS、コミット完了。
  - 機能ループ: 全タスク done、機能全体レビュー PASS、**要件↔設計↔実装トレーサビリティ PASS**（FR-5.21.2）、**機能実機検証 PASS**（FR-5.3）、設計書 staleness PASS、デッド資産 PASS、タスク-設計書紐付け PASS、attestation 出力、コミット・プッシュ完了。
  - 修正ループ: 機能ループに準じる。加えて**要件↔影響範囲↔実装トレーサビリティ PASS**（FR-5.21.3）と影響範囲分析の結果が attestation に記録されていること。
  - システムループ: 全機能ループ完了、**S6.5 要件↔設計カバレッジ PASS**（FR-5.21.1）、全ペルソナ駆動テスト完了、システム全体スモーク PASS、顧客向けドキュメント整備完了。
- **FR-5.2.6 黙った代替の禁止**: 実機検証を Robolectric / プレビュー / mockで代替してクローズ条件を満たしたことにしてはならない。代替する場合は明示的にフラグを立て、理由と影響を `autodev/state/loops/<対象名>.json` に記録し、ループは閉じない（人手承認待ち状態）。
- **FR-5.2.7 ループ間の状態共有**: ループは状態ファイル（`autodev/state/`）を経由してのみ前後ループと情報を受け渡す。AI セッションメモリだけに依存する状態は持ってはならない。

### 5.3 実機検証

- **FR-5.3.1 実機検証の対象**: 1機能ループ・1修正ループのクローズ条件として、以下のいずれか1つ以上の実機検証を必ず実施する。
  - **画面**: Maestro / Compose UI Test で実機（エミュレータ/実機）を起動し、対象画面をレンダリングし、想定された UI 要素・操作結果が得られることを検証する。スクリーンショットを `autodev/evidence/screenshots/<対象名>/` に保存する。
  - **API**: 外部 API 連携がある場合、実 HTTP リクエスト（または MockWebServer を使った単体検証 + 実機での疎通確認）で想定レスポンスが返ることを検証する。通信ログ（ApiLog）を logcat 証跡として保存する。
  - **DB**: 本番初期化経路で構築した実 SQLite（または MySQL）に対して round-trip（INSERT → SELECT → UPDATE → DELETE）が想定通り動作することを検証する。
- **FR-5.3.2 実機検証の前提条件**: 実機検証実行前に `autodev/scripts/healthcheck.sh` が PASS することを必須とする。dev 環境（JDK/SDK、Gradle、デバイス/エミュレータ、アプリ起動マーカー、Maestro）の健全性が機械実証されていない状態で実機検証に進んではならない。
- **FR-5.3.3 fallback 禁止**: ヘルスチェックは fail を fail として扱う。SDK/JDK 不在を別環境で代替したり、デバイス未接続をスキップで通過させたり、実機検証を Robolectric/mock で代替したりしてはならない。
- **FR-5.3.4 証跡保存**: 実機検証の証跡（スクショ、HTTP ログ、DB ダンプ、Maestro / Compose UI Test トレース）は必ず保存し、機能・修正ループ完了報告に含める。
- **FR-5.3.5 失敗時の挙動**: 実機検証で fail が出た場合、AIエージェントは fail 内容を入力としてイテレーションループに戻り、修正→再検証する。最大試行回数は `autodev/config/loop.yaml` の `max_retry_per_loop`（既定: 10）で制御し、超過時は人手介入待ちで停止する。

### 5.4 設計書同期

- **FR-5.4.1 設計書の役割**: 設計書は「実装の地図（下流）」であり、契約（上流）ではない。実装が変わったら設計書を実装に合わせて更新する。ただし機能開発前の「全体設計書」と、システムループ S6.5（FR-5.21.1）で先行作成される各機能の `docs/設計/features/<機能名>.md` は、未実装段階の指針として新規作成されることがあり、開発進行に伴い実装が成立した時点でそれが正となる。
- **FR-5.4.2 設計書の構造**: 設計書は階層的・薄く・ポインタ集として `docs/設計/` 配下に配置する（顧客提供物の一部）。
  - `docs/設計/README.md`: 全体像、重要な落とし穴、不変条件のみ。
  - `docs/設計/architecture.md`: アーキテクチャ概要、ディレクトリ地図。
  - `docs/設計/全体設計書.md`: システム全体の機能間関係、画面遷移、データフロー、共通機能（システムループ S4 で AI が新規作成）。
  - `docs/設計/data_model.md`: DB スキーマ概要（実 DB から生成）。
  - `docs/設計/api.md`: API IF 概要（実ルーティングから生成）。
  - `docs/設計/features/<機能名>.md`: 各機能のローカル規約・データモデル・IF（機能ループ F2 で新規作成 or 既存読込、F7 で同期）。
- **FR-5.4.3 同期タイミング**: イテレーションループの「設計書同期」ステップ（I2）で、変更箇所に対応する設計書セクションを実装から再生成または差分同期する。
- **FR-5.4.4 staleness 検出**: 設計書と実装の乖離は機械的に検出する。最低限以下を実装する。
  - DB スキーマ設計書 vs 実 DB スキーマの diff。
  - API 設計書 vs 実 Retrofit interface の diff。
  - 乖離検出は `autodev/scripts/check_design_sync.sh` として提供し、機能・修正ループのクローズ前ゲートとして実行する。
- **FR-5.4.5 設計書の上限**: 設計書のサイズには上限を設ける。1ファイル 500 行を超えそうな場合は分割を促す。

### 5.5 テスト戦略

- **FR-5.5.1 テストスコープの最適化**: テストは変更領域に限定する。具体的には以下。
  - (a) 変更ファイルが属するディレクトリのユニットテスト。
  - (b) 変更により影響を受ける（import チェーン上にある）モジュールのユニットテスト。
  - (c) 軽量な全体スモーク（assembleDebug、アプリ起動（APP_STARTED）、ホーム画面表示）。
  - 網羅テスト（全スイート実行）は別途定義する「テストフェーズ」でのみ実施する。
- **FR-5.5.2 テストの構成**:
  - バックエンド: JUnit（`app/tests/`、`test_unit.sh` 経由）。
  - フロントエンド ユニット: Robolectric/JVMテスト。**フロントエンドの実機検証の代替には使ってはならない。**
  - E2E: Maestro / Compose UI Test（`maestro/`、`run_maestro.sh` 経由）。
- **FR-5.5.3 self-bootstrap テスト禁止**: テストが自分でスキーマを作って自分で検証する形は禁止する。実機検証は実アプリの初期化経路（APK インストール→起動→Room 自動初期化）で構築した実 DB / 実アプリに対して走らせる（単体テストの in-memory Room はこの限りではない）。
- **FR-5.5.4 テストの「緑」の意味の監査**: 各テスト実行ごとに、収集エラー数、skip / xfail の件数とその理由、重要モジュールのカバレッジ下限を記録する。pass 数だけで判定しない。
- **FR-5.5.5 ground-truth 再実行**: AIエージェントの自己申告は信用しない。ハーネス側で同じテストを再実行し、結果の乖離を検出する。

### 5.6 状態管理・進捗管理

- **FR-5.6.1 状態ファイルの場所**: すべての状態ファイルは `autodev/state/` 配下に配置し、Git で tracked とする。
- **FR-5.6.2 状態ファイルの粒度**: 1ファイル数 KB 〜 数十 KB を上限の目安とする。巨大な単一 plan.json は禁止。機能別・ループ別に分割する。
- **FR-5.6.3 atomic 更新**: 状態ファイルの更新は atomic に行う（一時ファイル書き出し→rename）。
- **FR-5.6.4 監査可能差分**: 状態ファイルの変更は人間が diff で読める形式（JSON / YAML、整形済み）で保存する。
- **FR-5.6.5 状態ファイルの主要種類**: 詳細スキーマは付録B。
  - `autodev/state/loops/system.json`: システムループ全体の状態。
  - `autodev/state/loops/<対象名>.json`: 機能ループまたは修正ループの状態。
  - `autodev/state/tasks/<対象名>.json`: タスク分割・設計書セクション紐付け・進捗。
  - `autodev/state/iterations/<対象名>/<イテレーション番号>.json`: 各イテレーションの記録。
  - `autodev/state/attestations/<対象名>.json`: 完了の証拠付き attestation。
  - `autodev/state/personas/personas.json`: ペルソナ定義。
  - `autodev/state/persona_tests/<実施番号>.json`: ペルソナ駆動テスト実施記録。
  - `autodev/state/history/`: 進捗.md 旧スナップショット・PreCompact スナップショット。
- **FR-5.6.6 attestation**: ループの完了は、何を・どの実機検証で確認したかを記述した attestation を必ず伴う。証拠（スクショ、ログ）へのパスを含む。証拠なき完了は無効。
- **FR-5.6.7 開発進捗状況.md の自動同期**: `autodev/state/` 配下のいずれかの JSON ファイルが書き込まれるタイミングで、必ず `autodev/開発進捗状況.md` を再生成または差分更新する。状態 JSON の書き込みと進捗.md の更新は atomic に近い形で連続実行され、片方だけが更新された状態でAIエージェントが次のステップに進むことを禁止する（pre-commit hook で検出）。進捗.md の構造は付録D 参照。生成スクリプトは `autodev/scripts/update_progress_md.sh`。一貫性チェックは `autodev/scripts/check_progress_md_sync.sh`。
- **FR-5.6.8 進捗.md の人間可読性要件**: 純粋な JSON ダンプではなく、人間が日本語で読んで状況を理解できる文章 + 表 + 簡潔な箇条書きで構成する。機能リストは ✅ / 🔄 / ⏳ などで状態を視覚化し、数値（完了率、経過時間、リトライ回数）を1行サマリで提示する。「次に AI が行うアクション」を具体的な手順1〜3行で記述する。

### 5.7 ログ・証跡

- **FR-5.7.1 ログの保存先**: AIエージェントの出力ログ、ツール実行ログ、検証ログは `autodev/logs/<日付>/` 配下に保存する。
- **FR-5.7.2 ログのローテーション**: 30日経過したログは `autodev/logs/archive/` に圧縮アーカイブする。
- **FR-5.7.3 証跡の保存先**: 実機検証・ペルソナ駆動テストの証跡は `autodev/evidence/` 配下に対象別に保存する。
- **FR-5.7.4 ログとコミットの紐付け**: 各コミットメッセージには、対応するイテレーション ID（例: `iter:<対象名>:<番号>`）を含める。

### 5.8 エラー・障害復旧

- **FR-5.8.1 冪等再開**: AIエージェントセッションが何らかの理由で中断した場合、ユーザーが再度プロンプトを再投入することで、状態ファイルから直前のループ位置を復元して再開できる。AI は起動直後に `autodev/開発進捗状況.md` を最初に読み込んで前回の状況を把握する。
- **FR-5.8.2 進捗ゼロの検出**: ループが進捗ゼロのまま一定時間（既定: 30分）経過した場合、AIエージェントは現状報告を出して人手介入待ちで停止する。
- **FR-5.8.3 環境破損時の挙動**: ヘルスチェック fail を検出した場合、AIエージェントは修復を試みる（`bash setup.sh` 再実行（冪等）、エミュレータ再起動、アプリ再インストール）。修復試行回数上限（既定: 3回）に達した場合は人手介入待ちで停止する。
- **FR-5.8.4 ロールバック**: イテレーションのコミットが事後的に問題ありと判断された場合、AIエージェントは `git revert` で個別イテレーションを取り消すことができる。force push は禁止。
- **FR-5.8.5 デッド資産の検出**: 機能・修正ループ完了前に、誰も使わない表/列/モジュール、誰も作らない使用資源を gate で検出する（`autodev/scripts/check_dead_assets.sh`）。

### 5.9 ペルソナ駆動テスト

- **FR-5.9.1 ペルソナ定義**: AI は複数ペルソナ（既定: 3〜5体）を `autodev/state/personas/personas.json` に定義する。各ペルソナは「年齢／職業／IT リテラシー／スマートフォン利用文脈（通勤中・屋外・片手操作等）／優先度／制約条件」を含む。定義タイミングは次のいずれか:
  1. システムループの S3。
  2. 機能ループ F6.5 または修正ループ R6.5 のペルソナテスト発動時で、かつ `personas.json` が不在 or 最低人数を下回るとき（フォールバック）。
  事前に `autodev/scripts/ensure_personas.sh` を実行し、exit 0 ならそのまま再利用する。**既存定義は再生成・上書きしない**。不足時のみ ID 末尾追加で差分追加する（機能ループ／修正ループで生成された定義をシステムループが消さないため）。フォールバックの入力は scope に応じて次のとおり: scope=feature は機能要件書 + 既存設計書、scope=fix は修正対応書 + R2 の `impact_analysis` + 影響範囲の設計書。
- **FR-5.9.2 ペルソナテスト起動条件**: AI は次の3スコープで必ずペルソナテストフェーズを起動する。
  1. **scope=system**: システムループにおいて、3機能ループの完了ごと、および全機能完了後（S7 / S9）。機能数が3未満の場合は全機能完了時のみ実施。
  2. **scope=feature:<機能名>**: 機能ループの F6.5（機能単位開発モード、およびシステムループ S9 内の各機能ループ）。`autodev/config/loop.yaml::feature_loop.persona_test_enabled=true`（既定）のときに発動する。FR-5.9.8 参照。
  3. **scope=fix:<修正名>**: 修正ループの R6.5。`autodev/config/loop.yaml::fix_loop.persona_test_enabled=true`（既定）かつ修正対応書の `severity` が `fix_loop.persona_test_severity_threshold`（既定 `low`）以上のときに発動する。ペルソナテスト由来の改善要望に対応する修正（フロントマター `source_scope` が `feature` / `fix`）では再帰的な R6.5 発動はスキップする。FR-5.9.9 参照。
- **FR-5.9.3 ペルソナテストの内容**: AI が各ペルソナをロールプレイし、ペルソナが実行しそうな典型シナリオを Maestro / Compose UI Test で実機（エミュレータ/実機）上で実行する。シナリオは「目的達成型」「迷い・誤操作型」「エラーケース型」の3類を最低含み、**Android 固有操作（戻るボタン / 画面回転 / バックグラウンド退避→復帰 / プロセス kill 後の状態復元 / 機内モード / 権限拒否）を最低1つ織り込む**。flow 内の要所で `takeScreenshot` を取り、改善要望抽出時の画面キャプチャを確実に得る。
- **FR-5.9.4 改善要望の自動生成**: ペルソナテスト終了後、各ペルソナの視点で「使いにくかった点」「不足機能」「誤解しやすい表現」「Android 作法（戻る・回転・通知）への違反」を AI が列挙し、各項目を `autodev/inputs/修正対応/修正_<改善名>.md` として自動生成する。**機能ループ F6.5 / 修正ループ R6.5 で生成する場合は UI 観点（視覚階層・コントラスト・タップ領域・ローディング/エラー表現・誤操作リカバリ等）も同時に吸収する**（システムループでは S11 `autodev-ui-polish` が別途走るが、機能/修正ループでは S11 が走らないため）。ファイル名規約は次のとおり:
  - scope=system: `修正_<改善名>.md`
  - scope=feature: `修正_persona_feature_<機能名>_<改善名>.md`
  - scope=fix: `修正_persona_fix_<修正名>_<改善名>.md`
  各ファイルのフロントマターには `source_scope` / `source_target` / `severity` を必ず記載する。
- **FR-5.9.5 改善要望の対応**: 生成された修正対応ファイルは、優先度順に修正ループで順次対応する。優先度ジレンマがある場合は最小限の質問を発する。**機能ループ F6.5 / 修正ループ R6.5 由来の改善要望は、それぞれ F7 / R7 クローズ前にすべて消化する**（持ち越さない）。
- **FR-5.9.6 ペルソナテスト証跡**: ペルソナごと・シナリオごとのスクショ・logcat・改善要望リストを `autodev/evidence/persona_tests/<実施番号>/` に保存する。スコープ別の証跡は `<実施番号>/feature_<機能名>/` `<実施番号>/fix_<修正名>/` のサブディレクトリに格納する。
- **FR-5.9.7 ペルソナテストの記録**: `autodev/state/persona_tests/<実施番号>.json` に次を記録する: `scope`（`system` / `feature` / `fix`）/ `target`（機能名 or 修正名 / system のときは省略可）/ `triggered_from`（`S7` / `S9` / `F6.5` / `R6.5`）/ 実施日時 / 対象機能群 or 影響範囲 / 参加ペルソナ / 生成された改善要望ID一覧 / 対応結果 / `blocking_issues` / `result` / `cycles_used` / `cycles_max`。呼び出し元の attestation（機能 / 修正）には `persona_test_ref: <実施番号>` を併記する。**採番ルール（全 scope 通し連番）**: `test_number` は scope=system / feature / fix すべて同じディレクトリを共有するため、新規発番時は AI が既存ファイル群の `.test_number` を全件読んで `max + 1` を採番する（初回は 1）。これにより F6.5 / R6.5 / S7 / S9 が並行発生しても採番が衝突しない。
- **FR-5.9.8 機能ループ F6.5 のペルソナテスト**: 機能ループは F6（受け入れ条件の実機検証）完了後、F7（クローズ）前に必ず F6.5 として `scope=feature:<機能名>` のペルソナテストを実施する。シナリオは「当該新機能を主動線とし、既存連携動線も含める」。改善要望はサブ修正ループで F7 クローズ前にすべて消化する。`loop.yaml::feature_loop.persona_test_max_cycles`（既定 2）を超えた残課題は `blocking_issues` に集約し `blocked=true` で停止（F7 へ進まない）。`feature_loop.persona_test_enabled=false` のときは F6.5 をスキップしてよい（規約15 ホワイトリスト対象 / AI 判断による省略は禁止）。F7 クローズゲートに `autodev/scripts/check_persona_test_evidence.sh <機能名> feature` を含める。
- **FR-5.9.9 修正ループ R6.5 のペルソナテスト**: 修正ループは R6（影響範囲の実機検証）完了後、R7（クローズ）前に必ず R6.5 として `scope=fix:<修正名>` のペルソナテストを実施する。シナリオは R2 の `impact_analysis`（files / screens / room_entities / dao / apis / permissions）を入力に「修正対象 + 影響範囲動線」を中心とし、リグレッション観点を必ず1類以上含める。改善要望はサブ修正ループで R7 クローズ前にすべて消化する。`loop.yaml::fix_loop.persona_test_max_cycles`（既定 2）を超えた残課題は `blocking_issues` に集約し `blocked=true` で停止（R7 へ進まない）。`fix_loop.persona_test_enabled=false`、または修正対応書の `severity` が `fix_loop.persona_test_severity_threshold` 未満、または当該修正がペルソナテスト由来（フロントマター `source_scope` が `feature` / `fix`）の場合は R6.5 をスキップする（無限ループ防止）。スキップ時は `state/persona_tests/<番号>.json` に `r6_5_skipped: true` / `r6_5_skip_reason: <理由>` を記録（判定主体は AI）。R7 クローズゲートに `autodev/scripts/check_persona_test_evidence.sh <修正名> fix` を含める。

### 5.10 全体設計書

- **FR-5.10.1 全体設計書の作成**: システムループの S4 で AI が `docs/設計/全体設計書.md` を新規作成する。要件定義書とペルソナ定義を入力とし、システム全体の機能間関係、画面遷移、データフロー、共通機能（認証基盤、エラー処理基盤、ロギング基盤など）、共通スタイル方針を整理する。
- **FR-5.10.2 全体設計書の役割**: 「機能リスト補完」「機能ループの実装方針検討」「ペルソナテスト時の動線確認」の指針として参照される。開発進行に伴い実装が成立した部分は実装が正となり、全体設計書は実装に同期される。
- **FR-5.10.3 機能リスト補完への利用**: システムループの S5 で、全体設計書から共通機能・基盤機能・データフロー上必要な機能を抽出し、機能リストに追加する。要件定義書に明示されていなくても、全体設計上必要な機能を漏らさず開発対象とする。

### 5.11 修正対応モード

### 5.21 要件↔設計↔実装 トレーサビリティ確認

要件定義書 ⇒ 設計書 ⇒ 各機能の実装、と進行する開発プロセスにおいて、機能完成後に「実は要件が漏れていた」「設計書に書いていた要素が実装されていなかった」と判明する事故を防ぐ。3スコープ（システム / 機能 / 修正）でカバレッジを機械検査し、ギャップ検出時はループ内の追加タスクで吸収する。Android 固有の `layer`（ui / viewmodel / domain / data / test）・testTag・Room Entity・Maestro flow / Compose UI Test を活用して、要件項目ごとに対応する実装層・実装ファイルの存在を検査する点が Web 版との大きな違い。

#### 5.21.2 機能ループ F5.5「要件⇄設計⇄実装 トレーサビリティ照合」

- **FR-5.21.2.1 起動条件**: 機能ループ F5（機能全体レビュー）の完了後、F6（機能実機検証）に進む前に skill `autodev-coverage-check` を `scope=feature:<機能名>` で発動する。`loop.yaml::feature_loop.traceability_enabled=false` の場合のみスキップ可能（既定 true / 規約15 ホワイトリスト対象）。
- **FR-5.21.2.2 4層トレーサビリティの検査**: `autodev/scripts/check_traceability.sh <機能名> --scope feature` を実行し、以下4層の連鎖を機械検査する。
  - **要件項目**: 機能要件書（または要件定義書内の機能セクション）の「画面要素 / ユーザー操作 / エラーケース / データモデル / 受け入れ条件」リスト項目
  - **設計セクション**: `docs/設計/features/<機能名>.md` の対応セクション
  - **タスク**: `state/tasks/<機能名>.json::tasks[]` の `design_doc_ref.section` および `layer`
  - **実装ファイル**: `state/iterations/<機能名>/*.json::files_changed[]` および リポジトリ内 grep で実体確認
- **FR-5.21.2.3 layer × カテゴリ マッピング（Android 固有）**: 各要件カテゴリに対して期待される `layer` 集合を以下に定義し、タスクの `layer` が期待集合のいずれかにマッチするかを検査する。`expected_layers=[]` の場合は layer 検査スキップ（layer_match=true 固定）。
  - 画面要素 → `[ui]` + extra: testTag 候補抽出 → `app/src/main/.../ui/` で `Modifier.testTag("候補")` が grep でヒットするか
  - ユーザー操作 → `[ui, viewmodel]`
  - エラーケース → `[ui, viewmodel, domain]`
  - データモデル → `[data]` + extra: 対応 Room Entity Kotlin ファイル (`app/src/main/.../data/`) の `class|object|@Entity` 宣言を grep
  - 受け入れ条件 → `[test]` + extra: `maestro/*.yaml` または `app/src/androidTest/*Test.kt` のいずれかで対象機能名 / testTag 参照あり
  - API → `[data, domain]`
  - **影響範囲 → `[]`（layer 不問・主に scope=fix で発火）** + extra: 項目に含まれるファイルパス/ファイル名 token を `app/src/` と `maestro/` で grep
- **FR-5.21.2.4 `ui_required: false` 機能の例外**: `system.json::features[<機能名>].ui_required == false` のとき、`extract_requirement_items` が**「画面要素」「ユーザー操作」カテゴリを抽出対象から除外**する（DB 基盤など UI なし機能の正常パターン）。要件書にこれらの項目が記載されていても items[] に含まれないため、誤ギャップを生まない。jq の boolean 判定は `(.ui_required == false)` で行う（`// "true"` 演算子は false 値を null と同じく置換するため、boolean を直接比較する）。
- **FR-5.21.2.5 AI による補完判定**: スクリプトは英数字トークン・testTag 候補・ファイル名の grep で判定するため、日本語のみで記述された要件項目は false-negative になりうる。AI は出力 `state/traceability/<機能名>.json::items[]` を点検し、`covered=false` 項目について実装の実体（Compose Composable / ViewModel / UseCase / Repository / Room Entity / Maestro flow / Compose UI Test）を直接確認する。false-negative なら `items[].covered_by_ai=true` + `items[].ai_reason="<根拠>"` を JSON に追記する（ground-truth 補完 / FR-5.5.5 と同じ思想）。`ai_reason` は空文字列禁止（NFR-6.3.1 監査性）。**「TODO として残す」「将来対応」「次セッション」等の言い訳語を含めることも禁止（規約14 違反）**。`check_traceability.sh` は次回起動時に既存 JSON の `covered_by_ai` / `ai_reason` を `(category, item)` 一致で承継するため、追加タスク実装後の再走査でも AI 判定は維持される。
- **FR-5.21.2.6 ギャップ解消**: 真のギャップは追加タスクとして `state/tasks/<機能名>.json::tasks[]` に push する。各追加タスクは以下を必須とする。
  - `id`: 既存 max + 1
  - `title`: ギャップ要件項目を1文で記述
  - `design_doc_ref`: `{ "file": "docs/設計/features/<機能名>.md", "section": "<対応セクション>" }`（既存スキーマ準拠 / 規約5）
  - `layer`: 期待 layer 集合のいずれか（Android 規約 / B.3 既存スキーマ）
  - `source`: `"traceability_gap"`
  - `cycle`: 検出された `cycle` 値
  - `status`: `"pending"`
- **FR-5.21.2.7 F4 イテレーションループへの差し戻し**: 真のギャップがある場合、追加タスクを F4 イテレーションループで直列消化 → F5 全体レビュー → F5.5 再走査 のサイクルを回す。上限は `loop.yaml::feature_loop.traceability_max_cycles`（既定 2）。超過時は残ギャップを `state/loops/<機能名>.json::blocking_issues` に集約し `blocked=true` で停止する（F6 へ進まない / 規約12 準拠）。
- **FR-5.21.2.8 F7 クローズゲート**: F7 クローズ手順に `check_traceability.sh <機能名> --scope feature --gate` を追加する（既存 `check_testtag_coverage.sh` の直後・`check_real_device_evidence.sh` の前）。最新 `tasks` を反映した最終確認として機能する。
- **FR-5.21.2.9 attestation 記録**: F7 attestation（B.5）に `traceability_ref` フィールド（`state/traceability/<機能名>.json` のパス）と `traceability_passed` フィールド（boolean）を含める。`traceability_enabled=false` の場合は `traceability_passed=true` / `traceability_ref=null`（SKIP も PASS 扱い）とする。

#### 5.21.3 修正ループ R5.5「要件⇄影響範囲⇄実装 トレーサビリティ照合」

- **FR-5.21.3.1 起動条件**: 修正ループ R5（イテレーション）の完了後、R6（実機検証）に進む前に skill `autodev-coverage-check` を `scope=fix:<修正名>` で発動する。`loop.yaml::fix_loop.traceability_enabled=false` の場合のみスキップ可能（既定 true / 規約15 ホワイトリスト対象）。
- **FR-5.21.3.2 検査対象**: 修正対応書の受け入れ条件・影響範囲セクション、R2 で記録された `impact_analysis`（`files / screens / room_entities / dao / apis / permissions`）、`state/tasks/<修正名>.json`、`state/iterations/<修正名>/*.json::files_changed[]`。
- **FR-5.21.3.3 AI 補完・ギャップ解消・差し戻し**: F5.5（FR-5.21.2.3 〜 FR-5.21.2.7）と同じ手順・同じ layer マッピングを `scope=fix:<修正名>` で実施。追加タスクは `source: "traceability_gap"` で push し、R5 イテレーションループへ戻る。上限は `loop.yaml::fix_loop.traceability_max_cycles`（既定 2）。
- **FR-5.21.3.4 R7 クローズゲート**: R7 クローズ手順に `check_traceability.sh <修正名> --scope fix --gate` を追加する（F7 と同じ位置）。R7 attestation に `traceability_ref` / `traceability_passed` フィールドを含める。

#### 5.21.4 既存ゲートとの責務分離

| ゲートスクリプト | 検査範囲 | スコープ |
|---|---|---|
| `check_task_design_link.sh`（既存 / FR-5.2.3） | タスク↔設計書セクション網羅 | 機能・修正（F3 / R4 時点で実行） |
| `check_design_sync.sh`（既存 / FR-5.4.4） | 設計書↔実装 staleness（API・Room schema） | 全般 |
| `check_room_schema.sh`（既存） | Room schema export と Migration テストの整合 | 機能・修正（schema 変更時） |
| `check_testtag_coverage.sh`（既存） | 主要 UI 要素への testTag 付与網羅 | 機能・修正 |
| `check_requirement_design_coverage.sh`（新規 / FR-5.21.1） | 要件書↔設計書（機能セクション網羅 + 必須サブセクション + testTag 候補） | システム |

これらは重ならず補完関係になる。既存 `check_task_design_link.sh` は引き続き設計書セクションの「下流側」網羅を検査し、新規 `check_traceability.sh` は要件側からの「上流側」連鎖を Android 固有レイヤ（ui / viewmodel / domain / data / test + testTag + Maestro）で検査する。

#### 5.21.5 ループ無限化の防止と完了宣言ゲートとの連動

- `source: "traceability_gap"` 由来の追加タスクからは新たな `traceability_gap` を派生させない（AI は同 source タグのタスクに対して F5.5/R5.5 のギャップ判定を抑制する）。
- 各スコープに `max_cycles` を必須化し、超過時は必ず `blocked=true` で停止する（人手介入が必要 / 規約12 準拠）。
- `check_system_completion.sh` は完了宣言の絶対前提として、以下も検査する:
  - `system.json::s6_5_result.passed == true`（`coverage_check.enabled=false` のときを除く）
  - 全機能 attestation の `traceability_passed == true`
  - 全修正 attestation の `traceability_passed == true`
- AI 判定（`covered_by_ai`）の濫用防止のため、`items_with_ai_pass` の件数を `state/traceability/<対象名>.json` で公開し、肥大化シグナルは `HARNESS_NOTES.md` に記録する。

### 5.13 Gradle / Android SDK 環境の運用

- **FR-5.13.1 ラッパー経由実行の必須化**: ビルド・テスト・端末操作は必ず `autodev/scripts/` のラッパー経由で行う。素の `gradle` `./gradlew` `adb` `emulator` `avdmanager` `sdkmanager` `maestro` `apksigner` を直接実行することを禁止する。
- **FR-5.13.2 環境の自動セットアップ**: `setup.sh` →（内部で）`autodev/scripts/setup_android_env.sh` が以下を無人実行する（冪等・非対話・sudo 非依存。版数は `autodev/config/android_env.yaml` にピン留め）。
  1. JDK 17（Temurin）をユーザー領域（`~/.autodev-android/jdk`）へ導入（システムに 17 があれば再利用）。
  2. Android cmdline-tools を取得し、`sdkmanager` で platform-tools / platforms / build-tools / emulator / system-image を導入（`yes | sdkmanager --licenses` で自動同意）。
  3. AVD を作成（`avdmanager`。hw.keyboard=yes 等の安定化設定込み）。
  4. Gradle wrapper が未生成なら Gradle 配布版を取得して `gradle wrapper` で生成。
  5. `local.properties`（sdk.dir）を自動生成。
- **FR-5.13.3 ラッパースクリプトの提供**: AIエージェントが誤って素のコマンドを呼ばないよう以下を提供する。
  - `run_gradle.sh <task...>`: JAVA_HOME/ANDROID_SDK_ROOT を解決し、タイムアウト・ログ記録付きで `./gradlew` を実行。
  - `run_adb.sh` / `run_emulator.sh` / `run_sdkmanager.sh` / `run_maestro.sh`: 同様の低レベルラッパー。
  - 高レベル: `build.sh` `install.sh` `uninstall.sh` `run.sh` `build_install_run.sh` `screenshot.sh` `logcat.sh` `logcat_error.sh` `clean_logs.sh` `test_unit.sh` `test_ui.sh` `test_maestro.sh` `start_emulator.sh` `stop_emulator.sh`。
  - 環境不足を検出した場合は `bash setup.sh` の実行（冪等）を促す。
- **FR-5.13.4 ヘルスチェックでの確認**: `autodev/scripts/healthcheck.sh` は JDK/SDK/Gradle/デバイス（無ければエミュレータ自動起動）/アプリ起動マーカー（`AppLog: APP_STARTED`）/Maestro を確認する。
- **FR-5.13.5 規約の周知**: `app/CLAUDE.md` に「ビルド・ADB 操作は必ずラッパー経由」と明記する。
- **FR-5.13.6 hook での強制**: pre-commit hook と Claude Code の PreToolUse hook で、素のコマンド直接呼び出しを検出して禁止する。
- **FR-5.13.7 生成物のコミット禁止**: `.gitignore` で `build/` `local.properties` keystore 類を ignore、pre-commit hook でも tracked 化を検出して fail とする。

### 5.14 実機検証（Compose UI Test / Maestro）のセットアップと運用

- **FR-5.14.1 初期状態での実機検証セットアップ完了**: `setup.sh` 完了時点で、以下が成立している。
  1. Compose UI Test の依存（compose ui-test-junit4 / test runner）が `app/build.gradle.kts` に定義済み。
  2. Maestro CLI が導入済み（`~/.maestro`。版は `android_env.yaml` で固定可）。
  3. AVD が作成済みで、headless エミュレータが起動できる。
  4. `maestro/smoke.yaml`（起動スモーク）が PASS することが確認されている。
  5. ルート Compose で `testTagsAsResourceId = true` が有効（Maestro から testTag を参照可能）。
- **FR-5.14.2 ラッパースクリプトの提供**: AIエージェントは実機検証を以下経由で運用する。
  - `run_maestro.sh <args...>` / `test_maestro.sh <flow> [対象名]`: Maestro 実行（タイムアウト・証跡保存付き）。
  - `test_ui.sh [テストクラスFQCN]`: Compose UI Test（connectedDebugAndroidTest）実行。
  - `run_real_device_check.sh <対象名> [flow|FQCN]`: 高レベルラッパー。デバイス確保 → build → install → 起動 → 検証 → ADB スクショ → logcat ダンプ + クラッシュスキャン → 証跡パスを attestation に追記までを1コマンドで行う。
- **FR-5.14.3 セットアップ確認スクリプト**: `autodev/scripts/check_android_setup.sh` が JDK / SDK / AVD / gradlew / Maestro の存在を確認する。fail なら `bash setup.sh`（冪等）を再実行する。
- **FR-5.14.4 実機検証手順の標準化**: AIエージェントが実機検証時に踏む手順を以下に固定する。
  1. `autodev/scripts/healthcheck.sh` で dev 環境健全性を確認。
  2. 機能要件書または修正対応書の受け入れ条件から、Compose UI Test（`app/src/androidTest/`）または Maestro flow（`maestro/<対象名>.yaml`）を作成・更新（testTag 参照・座標タップ禁止）。
  3. `autodev/scripts/run_real_device_check.sh <対象名> [flow|FQCN]` を実行。
  4. pass の場合: スクショ + logcat が `autodev/evidence/` に保存される。fail の場合: `logcat_error.sh` で原因を調査しイテレーションループに戻す。
  5. attestation に証跡パス（`real_device_evidence`）が記録されたことを確認。
  6. 進捗.md を再生成。
- **FR-5.14.5 Robolectric / プレビュー / mock 代替の構造的禁止**: `autodev/scripts/check_real_device_evidence.sh` は機能ループ完了前に、対象機能名のスクショ（1枚以上）と logcat ダンプが存在し、logcat にクラッシュ痕跡（FATAL/ANR）が無いことを確認し、満たさなければ fail。
- **FR-5.14.6 規約の周知**: `maestro/CLAUDE.md` に Maestro 運用規約（testTag 参照・座標禁止・appId 一致）を明記する。
- **FR-5.14.7 ヘッドレス既定 / ヘッドフル切替**: エミュレータは既定 headless（-no-window）。`HEADFUL=1` 環境変数で GUI 表示に切り替えられる。AIエージェント自身が勝手にヘッドフルを既定にすることは禁止。

### 5.15 Claude Code hooks の活用

- **FR-5.15.1 hooks 宣言場所**: `.claude/settings.json` にて hooks を宣言する。実体スクリプトは `autodev/hooks/claude/` 配下に置き、`.claude/settings.json` からは `$CLAUDE_PROJECT_DIR/autodev/hooks/claude/<name>.sh` を呼ぶ形にする（実行時 cwd は不定なので絶対パス起点が必須）。
- **FR-5.15.2 必須 hooks**: 最低限以下を実装する。
  - **SessionStart**: セッション開始時に `autodev/開発進捗状況.md` を AIエージェントに読み込ませる initial context として注入する。冪等再開時に前回状況を即把握できるようにする。
  - **UserPromptSubmit**: 起動プロンプトを受信した直後に、入力ファイルの形式チェックと起動モード判定を実行する。
  - **PreToolUse（matcher: Bash）**: AI が Bash ツールで素の `gradle` / `gradlew` / `adb` / `emulator` / `avdmanager` / `sdkmanager` / `maestro` / `apksigner` を直接呼ぼうとした場合に fail させ、ラッパーの使用を促す。
  - **PostToolUse（matcher: Edit|Write）**: AI が `autodev/state/` 配下の JSON ファイルを書き込んだ後に、`autodev/scripts/update_progress_md.sh` を自動実行する。
  - **PostToolUse（matcher: Bash, command: git commit）**: AI がコミット直後に、軽量ゲートを自動実行する。
- **FR-5.15.3 hooks の冪等性**: すべての hook スクリプトは冪等に設計する。
- **FR-5.15.4 hooks のログ**: 各 hook の実行ログは `autodev/logs/<日付>/hooks_<hook名>.log` に追記される。
- **FR-5.15.5 hooks のエスケープハッチ**: 環境変数 `AUTODEV_DISABLE_HOOKS=1` が設定されている場合のみ、すべての hook を no-op で通過させる（緊急時のみ。使用履歴を `HARNESS_NOTES.md` に記録する）。
- **FR-5.15.6 hooks 自体の棚卸し**: `autodev/HARNESS_NOTES.md` に各 hook の存在理由と直近の動作確認日を記録する。3〜6ヶ月毎に棚卸し。

### 5.16 Claude Code skills の活用

- **FR-5.16.1 skills 配置**: `.claude/skills/<skill名>/SKILL.md` の形で配置する。実体は `autodev/skills/<skill名>/` に置き、`.claude/skills/` からはシンボリックリンクまたは `autodev/scripts/sync_claude_dir.sh` によるコピーで参照する。Windows ネイティブ環境では symlink の作成権限が制限される場合があるため、WSL 利用を既定とし、Windows ネイティブで運用する場合はコピー方式に切り替える（sync_claude_dir.sh のオプション `--copy` で挙動切替）。
- **FR-5.16.2 必須 skills**: 最低限以下を実装する。
  - **autodev-system-loop**: システムループ S1〜S15 の標準手順。
  - **autodev-feature-loop**: 機能ループ F1〜F7 の標準手順。
  - **autodev-fix-loop**: 修正ループ R1〜R7 の標準手順。
  - **autodev-persona-test**: ペルソナ駆動テストの標準手順。
  - **autodev-integration-verify**: システムループ S10（全体結合確認・改善）の標準手順。
  - **autodev-ui-polish**: システムループ S11（UIデザイン改善）の標準手順。
  - **autodev-release-gate**: システムループ S14（デプロイ・リリース最終確認）の標準手順。
  - **autodev-gradle-env**: Gradle / ADB / エミュレータのラッパー経由実行の標準手順。
  - **autodev-device-verify**: Maestro / Compose UI Test による実機検証の標準手順。
  - **autodev-design-sync**: 設計書同期の標準手順。
  - **autodev-progress-md**: 進捗.md 再生成の標準手順。通常運用では PostToolUse hook が自動実行するため AI が手動発動する必要はないが、hook 失敗時の手動リカバリ・冪等再開時の整合確認などで AI が明示的に手順を辿る場合の補助 skill として位置づける。
  - **autodev-coverage-check**: システムループ S6.5 / 機能ループ F5.5 / 修正ループ R5.5 の要件↔設計↔実装トレーサビリティ確認の標準手順（FR-5.21）。
- **FR-5.16.3 SKILL.md の構成**: 各 SKILL.md は以下の構成を最低限満たす。
  1. front-matter（**標準フィールドは `name` と `description` のみ**。`description` にいつ・何のためにこの skill を使うかを簡潔に記述し、AI の発動判断材料にする。`version` 等の管理情報は HTML コメントで残す）
  2. 適用条件
  3. 標準手順（番号付き、各手順で呼ぶスクリプト）
  4. 完了条件
  5. 失敗時の挙動
  6. 関連 skills / スクリプト / 設計書へのポインタ
- **FR-5.16.4 SKILL.md の薄さ**: 各 SKILL.md は 300 行以内を目安とする。
- **FR-5.16.5 skills 同士の参照**: skill 間の参照は SKILL.md 内のリンクで明示する。
- **FR-5.16.6 skills のバージョニング**: 各 SKILL.md には版数とその変更履歴を記載する（HTML コメント可）。
- **FR-5.16.7 skills の更新と sync**: `autodev/skills/` の変更があった場合、`autodev/scripts/sync_claude_dir.sh` を実行して `.claude/skills/` に反映する。Sync スクリプトは pre-push hook にも組み込み、両者の乖離を検出する。

### 5.17 機能完了時の `/compact` 実行 【廃止】

**本要件（FR-5.17.x 全項）は廃止した。** `/compact` は Claude Code の **ユーザー入力専用スラッシュコマンド**で、
AIエージェントもフックも対話セッションでは発行できないため、「機能完了ごとの自動 `/compact`」は実現不可能だった。
コンテキスト逼迫時は Claude Code 組み込みの自動 compact が安全網として常時作動する。これに伴い compact 関連の
処理・定義（autodev-compact skill、PreCompact / Stop hook、`check_compact_executed.sh` ゲート、`compact_fix_*`
設定、attestation の `compacted_at`/`compact_status`）はツールから撤去済み。FR 番号は欠番として保持する。

  - キャッシュ可能なステップを活用。
- **NFR-6.1.6 起動時オーバーヘッド**: AIエージェント起動後、最初のプロンプト送信から最初のコード変更開始まで 5 分以内。
- **NFR-6.1.7 進捗.md 更新のオーバーヘッド**: 進捗.md の再生成は1回 100ms 以内を目標。差分更新を基本とし、フル再生成は機能完了時など要所のみ。
- **NFR-6.1.8 hook のオーバーヘッド**: 各 hook の実行は1回あたり 500ms 以内を目標。それを超える重い処理は非同期化または末尾処理で実行。

### 6.2 信頼性（無確認自動化）

- **NFR-6.2.1 無確認自動化の原則**: 起動プロンプト送信後、機能完了・修正完了・システム完了まで、AIエージェントはユーザーに承認を求めない。例外として以下の場合のみ最小限の質問を発する。
  - 要件定義書／機能要件書／修正対応書に解釈が一意に定まらない記述があり、AI が自律判断するとシステム全体方針に重大な影響を及ぼす場合。
  - 入力ファイルが見つからない、複数候補がある、フォーマット違反である場合。
  - 環境破損が修復試行上限に達した場合。
  - ループの最大試行回数を超過した場合。
  - ペルソナテストで生成された改善要望に優先度ジレンマがある場合。
- **NFR-6.2.2 ハングしない**: 進捗ゼロを検出したら停止して報告する。無限ループに陥らない。
- **NFR-6.2.3 dev 環境の常時健全性**: 各イテレーション開始時にヘルスチェック必須。fail なら修復、修復不能なら停止。
- **NFR-6.2.4 hook 失敗時の挙動**: 重要 hook（PreToolUse の禁止コマンド検出、PostToolUse の進捗.md 同期、PreCompact の状態保存）の失敗はループの停止条件とする。軽量 hook の失敗は警告に留める。

### 6.3 監査性

- **NFR-6.3.1 完了の追跡可能性**: 各機能・修正完了について、対象要件 → 変更コミット列 → 実機検証証跡 → attestation が一意に辿れる。
- **NFR-6.3.2 ペルソナテストの追跡可能性**: 各ペルソナテスト実施回 → 改善要望リスト → 修正ループ ID → 完了 attestation が一意に辿れる。
- **NFR-6.3.3 状態ファイルの可読性**: 状態ファイルは人間が直接 diff で読めるフォーマット。
- **NFR-6.3.4 ログの保持期間**: 30日間はオンラインで参照可能、それ以降は圧縮アーカイブ。
- **NFR-6.3.5 進捗.md による即時追跡可能性**: 状態 JSON 更新と進捗.md 更新のタイムスタンプ乖離は数秒以内に保つ。

### 6.4 保守性

- **NFR-6.4.1 DB 切替容易性**: SQLite と MySQL を環境変数の切替のみで入れ替え可能とする。
- **NFR-6.4.2 ハーネスの棚卸し**: ハーネス（環境ゲート、検証層、同期設計書、除外設定、各種規約、hooks 仕様）は定期棚卸しの対象。目安は Claude Code 本体の更新後、および 3〜6ヶ月毎。
- **NFR-6.4.3 ハーネス DRI**: ハーネスには説明責任者を1人置く（個人開発の場合は本人）。新たな回避策追加時はその理由を `autodev/HARNESS_NOTES.md` に記録する。
- **NFR-6.4.4 ツール本体の分離容易性**: `autodev/` と `.claude/` ディレクトリを削除するだけで、自動開発ツール関連が完全に消える状態を維持する。

### 6.5 セキュリティ

- **NFR-6.5.1 `--dangerously-skip-permissions` の前提**: 本ツールはAIエージェントがファイル操作・シェル実行を無確認で行うことを前提とする。dev 環境は使い捨て可能な開発環境であることを前提とする。本番環境上での使用は禁止する。
- **NFR-6.5.2 秘密情報の管理**: 署名 keystore は `autodev/secrets/`（setup.sh が自動生成）、SDK パスは `local.properties` で管理し、いずれも Git で ignore する。アプリが API キー等を必要とする場合は、ソースにハードコードせず `autodev/secrets/`（または `local.properties`）から Gradle 経由で BuildConfig に注入するか、ユーザー入力値として DataStore に保存する。AIエージェントはこれら秘密ファイルをコミットしてはならない（`.gitignore` + pre-commit hook で機械的に禁止）。
- **NFR-6.5.3 外部ネットワークアクセスの制限**: 外部 URL への破壊的な POST / DELETE 発行は想定していない。dev 環境からの不必要な外部通信は最小限とする。
- **NFR-6.5.4 顧客提供物への混入防止**: リリース手順書に「`autodev/` と `.claude/` を削除する」ステップを明示。`autodev/scripts/check_release_clean.sh` で機械検出。

### 6.6 コスト制約

- **NFR-6.6.1 追加料金が発生する起動形態の禁止**: `claude -p` をはじめ、Max プラン等で追加料金が発生する起動形態は使用してはならない。
- **NFR-6.6.2 セッション内消費トークンの最適化**: 不要なコンテキスト読み込みを避け、設計書を薄く保ち、領域別に追加ロードする運用を徹底する。
- **NFR-6.6.3 トークン消費の上限想定**: 1機能ループあたりのトークン消費は、組み込み自動 compact と進捗.md 薄さ維持により、シミュレーション上は 50K〜200K トークン程度を想定する。

## 第7章 技術スタック

### 7.1 AIエージェント実行環境

- **コマンド**: `claude --dangerously-skip-permissions`
- **モデル**: Claude（バージョンはユーザー環境に依存。`autodev/HARNESS_NOTES.md` に推奨バージョンを記録）
- **動作モード**: 対話モード。1セッション内で複数ループを連続実行する。
- **禁止コマンド**: `claude -p`、その他追加料金の発生する起動オプション。

### 7.2 アプリ UI 層

- **言語**: Kotlin（JDK 17）
- **UI**: Jetpack Compose（Material 3）。画面は `ui/screen/` の `*Screen.kt`、再利用部品は `ui/component/`、遷移は `ui/navigation/` の NavHost。
- **状態管理**: ViewModel + StateFlow / **非同期**: Kotlin Coroutines
- **テスト用識別子**: 主要UI要素（ボタン・入力欄・リスト・画面ルート）に `Modifier.testTag("<画面>_<要素>_<種別>")` を必ず付与（例: `login_email_input` `login_button` `home_add_button` `memo_save_button`）。
- **必須設定**: ルート Compose で `testTagsAsResourceId = true` を有効化（Maestro/UIAutomator から testTag を参照可能にする）。
- **配置**: `app/`（ルート直下、顧客提供物の一部）

### 7.3 アプリ データ層

- **DB**: Room（`data/local/`: AppDatabase / dao/ / entity/）。`exportSchema = true` とし、schema JSON を `app/schemas/` に git tracked。
- **設定保存**: DataStore（`settings/AppSettingsDataStore.kt` にキーを集約）。
- **API通信**: Retrofit + OkHttp（`data/remote/`。必要時のみ）。接続先 URL はハードコードせず DataStore 等から実行時解決。
- **ドメイン**: `domain/model/` + `domain/usecase/`。
- **単体テスト**: JUnit（`app/src/test/`。Room は in-memory、API は MockWebServer 可。ただし実機検証の代替には使わない）。
- **ログ**: `util/AppLogger.kt` 経由・統一タグ（`AppLog` `ApiLog` `DbLog` `UiLog`）。起動時に `APP_STARTED` マーカーを出力（healthcheck が確認）。

### 7.4 データベース管理

- **既定**: Room（端末内 SQLite）。サーバ DB は持たない。
- **スキーマ管理**: Room の version + `app/schemas/` の exportSchema 出力で一元管理。
- **マイグレーション**: schema version を上げたら `Migration` 定義 + `MigrationTestHelper` による androidTest が必須（`autodev/scripts/check_room_schema.sh` がゲート）。
- **初期化経路**: Room は初回起動時に自動初期化される。テスト・実機検証は実アプリの初期化経路（APK インストール→起動）で構築された DB に対して実行。

### 7.5 実機検証（E2E）

- **UIテスト**: Compose UI Test（`app/src/androidTest/`。testTag 操作・座標タップ禁止）。実行は `autodev/scripts/test_ui.sh`。
- **E2E**: Maestro（`maestro/*.yaml`。testTag 参照）。実行は `autodev/scripts/test_maestro.sh` / `run_maestro.sh`。
- **実行環境**: AVD エミュレータ（headless 既定。`HEADFUL=1` で GUI）または adb 接続実機。`autodev/scripts/start_emulator.sh` が boot 完了待ち・アニメーション無効化まで行う。
- **画面確認**: ADB screenshot（`autodev/scripts/screenshot.sh` → `autodev/evidence/screenshots/<対象名>/`）。
- **ログ確認**: adb logcat（`autodev/scripts/logcat.sh` / `logcat_error.sh` → `autodev/evidence/logcat/<対象名>/`。アプリ pid 絞り込み・クラッシュスキャン付き）。
- **高レベル検証**: `autodev/scripts/run_real_device_check.sh <対象名>`（build→install→起動→検証→スクショ→logcat→attestation 記録）。
- **代替禁止**: Robolectric / Compose プレビュー / mock 画面による実機検証の代替は禁止。

### 7.6 設計書/状態管理

- **設計書**: Markdown、`docs/設計/` 配下（ルート側、顧客提供物の一部。data_model / api / screen_flow / features/）
- **状態**: JSON、`autodev/state/` 配下、Git tracked
- **進捗ダッシュボード**: `autodev/開発進捗状況.md`（状態更新と同期して再生成）
- **ログ**: テキスト、`autodev/logs/` 配下
- **証跡**: PNG（ADB スクショ）/ logcat テキスト / テストレポート、`autodev/evidence/` 配下
- **成果物**: 署名済み APK、`autodev/artifacts/` 配下（世代保管）

### 7.7 補助ツール

- **静的解析**: Android Lint（`run_gradle.sh :app:lintDebug`。UnusedResources は `check_dead_assets.sh` が利用）
- **Git hook**: pre-commit / pre-push（設計書 staleness・進捗.md 同期・skills 同期・リリースクリーン）。設定は `autodev/hooks/` 配下。
- **シェル**: bash（Ubuntu 一次サポート / macOS ベストエフォート）。
- **Claude Code hooks**: `.claude/settings.json` で宣言。実体スクリプトは `autodev/hooks/claude/` 配下に bash で実装。
- **Claude Code skills**: `.claude/skills/<skill名>/SKILL.md` で配置。実体は `autodev/skills/<skill名>/`。
- **自動 compact**: Claude Code 標準機能（`/compact` はユーザー入力専用のためツールからは扱わない）。
- **ラッパー/高レベルスクリプト群**: `run_gradle.sh` `run_adb.sh` `run_emulator.sh` `run_sdkmanager.sh` `run_maestro.sh` / `build.sh` `install.sh` `uninstall.sh` `run.sh` `build_install_run.sh` `screenshot.sh` `logcat.sh` `logcat_error.sh` `clean_logs.sh` `test_unit.sh` `test_ui.sh` `test_maestro.sh` / `start_emulator.sh` `stop_emulator.sh` / `run_real_device_check.sh` `check_android_setup.sh` `check_real_device_evidence.sh` `check_room_schema.sh` `check_testtag_coverage.sh` `update_progress_md.sh` `check_progress_md_sync.sh` `sync_claude_dir.sh`。

### 7.8 ビルド・環境管理（ゼロタッチセットアップ）

- **ビルド**: Gradle（Wrapper 同梱・Kotlin DSL・Version Catalog）。素の `gradle`/`./gradlew` は禁止、`run_gradle.sh` 経由（タイムアウト・ログ・local.properties 自動整備付き）。
- **環境バージョンのピン留め**: JDK / cmdline-tools / platform / build-tools / system-image / AVD / Maestro の版数・取得URLは `autodev/config/android_env.yaml` に一元定義（スクリプトへのハードコード禁止）。
- **セットアップ**: `bash setup.sh` が JDK 17（Temurin → `~/.autodev-android/jdk`）・Android SDK（→ `~/Android/Sdk`、ライセンス自動同意）・AVD・Maestro・release 署名 keystore（→ `autodev/secrets/`）を**完全非対話・冪等・sudo 非依存**で自動導入し、初期実機検証（エミュレータでアプリ起動 + maestro/smoke.yaml + スクショ）まで無人で到達する。
- **パス解決**: ラッパーが ANDROID_SDK_ROOT / JAVA_HOME / PATH をスクリプト内で解決。ユーザーのシェル設定に依存しない。

### 8.6 初期状態

新規プロジェクト生成直後（`bash setup.sh` 完了後）の状態を以下とする。

- JDK 17 / Android SDK / AVD / Maestro / Gradle wrapper が `setup.sh` により自動導入済み（版数は `android_env.yaml`）。
- `app/` に最小の Android アプリ（Compose の Home 画面 + Room サンプル + DataStore + AppLogger）。`build.sh` で APK がビルドできる。
- 起動時に `AppLog: APP_STARTED` が logcat に出る（healthcheck の起動確認に使用。Web 版 /healthz 200 相当）。
- 主要UI要素に testTag（`home_root` `home_memo_input` `home_add_button` `home_memo_list`）が付与済み。ルートで `testTagsAsResourceId = true`。

### 9.3 機能単位開発フロー（機能ループ F1〜F7）

```

       ├─ 機能要件書および設計書の各セクションを走査し、対応するタスクを生成
       ├─ 各タスクに design_doc_ref（ファイルパス + 見出し or 行範囲）を必ず持たせる

(F4) 各タスクについてイテレーションループ（9.5）を直列実行
(F5) 機能全体レビュー
       （変更ファイル全体・関連設計書・タスクと設計書の対応を AI 自身が再点検）
       問題発見時は修正イテレーション追加
(F5.5) 要件⇄設計⇄実装 トレーサビリティ照合（autodev-coverage-check skill を scope=feature:<機能名> で発動 / FR-5.21.2）
       （loop.yaml::feature_loop.traceability_enabled=false のときはスキップ / 規約15 ホワイトリスト）
       ├─ check_traceability.sh <機能名> --scope feature --cycle <N>:
       │   - 要件項目（画面要素 / ユーザー操作 / エラーケース / データモデル / 受け入れ条件）

       │   - features[<対象>].ui_required=false の機能は「画面要素」「ユーザー操作」検査をスキップ

       → state/loops/<機能名>.json::f5_5_result.traceability_ref 更新 → 進捗.md 再生成
(F6) 機能実機検証（autodev-device-verify skill 発動）
       （F4 では変更領域のみ。F6 では機能要件書の受け入れ条件すべてを実機（エミュレータ/実機）で通す）
       fail なら fail 内容を入力に修正イテレーションへ戻る
(F6.5) ペルソナ駆動テスト（autodev-persona-test skill を scope=feature:<機能名> で発動 / FR-5.9.8）

### 9.4 修正対応フロー（修正ループ R1〜R7）

       ├─ 影響範囲の docs/設計/features/<関連機能名>.md を読込
       └─ 必要なら全体設計書の関連セクションを確認
(R4) タスク列に分割し autodev/state/tasks/<修正名>.json に記録
       ├─ 機能ループ F3 と同様、各タスクに design_doc_ref を持たせる
       └─ 影響範囲のすべての箇所がタスク化されていることを機械チェック
(R5) 各タスクについてイテレーションループ（9.5）を直列実行
(R5.5) 要件⇄影響範囲⇄実装 トレーサビリティ照合（autodev-coverage-check skill を scope=fix:<修正名> で発動 / FR-5.21.3）
       （loop.yaml::fix_loop.traceability_enabled=false のときはスキップ / 規約15 ホワイトリスト）
       ├─ check_traceability.sh <修正名> --scope fix --cycle <N>:
       │   - 修正対応書の受け入れ条件・影響範囲 ⇄ R2 の impact_analysis ⇄ tasks ⇄
       │     files_changed の4層連鎖を検査（F5.5 と同じ layer マッピング）
       │   - state/traceability/<修正名>.json に items[] / items_with_gap / passed を出力
       ├─ AI 補完判定: F5.5 と同様（covered_by_ai 補完 / 真のギャップは追加タスク）
       ├─ 真のギャップ有り → R5 イテレーションループへ戻る → R5.5 再走査
       └─ 上限 (fix_loop.traceability_max_cycles 既定 2) 超過 → blocking_issues 集約 →
          blocked=true 停止（R6 へ進まない / 規約12）
       → state/loops/<修正名>.json::r5_5_result.traceability_ref 更新 → 進捗.md 再生成
(R6) 修正全体レビュー + 影響範囲を実機（エミュレータ/実機）で実機検証
       ├─ 修正対応書の受け入れ条件を実機で通す
       ├─ 影響範囲の既存機能が壊れていないことを実機で確認
       └─ fail なら修正イテレーションへ戻る
(R6.5) ペルソナ駆動テスト（autodev-persona-test skill を scope=fix:<修正名> で発動 / FR-5.9.9）
       （次のいずれかでスキップ / 判定主体は AI:
          - loop.yaml::fix_loop.persona_test_enabled=false（規約15 ホワイトリスト）
          - 修正対応書の severity < fix_loop.persona_test_severity_threshold
          - 当該修正がペルソナテスト由来（source_scope が feature / fix / 無限ループ防止））

       ├─ ensure_personas.sh で既存 personas.json 再利用 or AI がフォールバック定義
       ├─ シナリオ: R2 の impact_analysis を入力に、修正対象 + 影響範囲動線
       │   （リグレッション観点を必ず1類以上含める / 各ペルソナ × 3類 + Android 固有操作）
       ├─ run_persona_tests.sh <番号> --scope fix:<修正名>（Maestro flow / takeScreenshot）
       ├─ 改善要望（UI 観点 + Android 作法も含む）→ 修正_persona_fix_<修正名>_<改善名>.md を自動生成

       │     （fix_loop.persona_test_max_cycles 上限 / 既定 2）

(R7) クローズ手順
      ├─ 機能ループ F7 と同じ各種ゲート
      │   （check_traceability.sh <修正名> --scope fix --gate（FR-5.21.3）/
      │    check_persona_test_evidence.sh <修正名> fix（FR-5.9.9） を含む）

```

### 9.5 1イテレーションループの詳細（I0〜I8）

```

  ↓
(I0) ヘルスチェック実行
      ├─ Android 環境チェック（JDK/SDK/AVD/Maestro）
      ├─ Gradle 実行可否
      ├─ デバイス/エミュレータ確保（無ければ自動起動）
      ├─ アプリ起動可否（APP_STARTED マーカー）
      └─ FAIL なら修復試行 → なお FAIL なら停止
(I1) 直列に1タスク分の小さな実装を dev 環境上で行う
      （design_doc_ref を参照しながら設計書のセクションと突き合わせて実装）
      （Python 実行は必ず run_gradle.sh / test_unit.sh 経由）
(I2) 設計書を実装から同期（変更箇所のみ）
      （autodev-design-sync skill 発動）
(I3) 変更箇所のコード・設計書レビューを AI 自身が実施。問題があれば修正
(I4) 変更領域に限定したテスト（unit + 影響範囲 + 全体スモーク）を実行
      ├─ backend は test_unit.sh 経由
      ├─ frontend は npm test
      └─ e2e は run_maestro.sh 経由
(I5) ground-truth 再実行（ハーネス側で同じテストを再実行し、申告との乖離を検出）
(I6) 設計書 staleness ゲート、デッド資産ゲート、タスク-設計書紐付けゲート、
     進捗.md 同期ゲートを実行
(I7) コミット（メッセージに iter:<対象名>:<番号> を含める）
      → PostToolUse hook が軽量ゲートを自動実行
(I8) autodev/state/iterations/<対象名>/<番号>.json 更新

### 9.7 検証スコープの最適化

- 1イテレーションでは、変更領域に限定したテストのみ実施する。`autodev/scripts/run_scoped_tests.sh <変更ファイル>...` が以下を判定して必要なものだけ走らせる。
  - 変更ファイルが `app/src/main/.../data/local/entity/` 配下 → そのモデルのテスト + Room Migration 整合確認 + そのモデルを使う routers のテスト
  - 変更ファイルが `app/src/main/.../data/remote/` 配下 → そのルータのテスト + e2e で当該エンドポイントを実 HTTP で叩くテスト
  - 変更ファイルが `app/src/main/.../ui/` 配下 → そのコンポーネントの Robolectric/JVMテスト（あれば）+ e2e で当該画面を実機（エミュレータ/実機）で開くテスト
  - 共通: 全体スモーク（assembleDebug 成功 / アプリ起動（APP_STARTED）/ ホーム画面表示）
- 機能ループ F6・修正ループ R6 では、要件書／修正書に列挙されたユーザーシナリオすべてを実機（エミュレータ/実機）で通す。
- システムループ S10 では、全 E2E スイートを走らせる（システム全体の網羅確認）。
- 「網羅テスト」を別フェーズで実行したいユーザーは `autodev/scripts/run_full_tests.sh` を手動実行する。

### 9.8 設計書同期の具体手順

イテレーションの (I2) で、変更ファイルに応じて以下を実行する。

| 変更領域 | 同期先設計書 | 同期手段 |
|---|---|---|
| `app/src/main/.../data/local/entity/` | `docs/設計/data_model.md` | 実 DB から `pragma table_info` 相当で取得しテーブル定義表を再生成 |
| `app/src/main/.../data/remote/` | `docs/設計/api.md` | Retrofit interface（@GET/@POST 等）から生成 |
| `app/src/main/.../domain/usecase/` | `docs/設計/features/<機能名>.md` | サービス一覧 + docstring を抜粋して同期 |
| `app/src/main/.../ui/` | `docs/設計/features/<機能名>.md` | コンポーネント一覧 + props 型を抜粋して同期 |
| `maestro/` | `docs/設計/features/<機能名>.md` | E2E シナリオ一覧を抜粋して同期 |
| 共通機能・基盤関連 | `docs/設計/全体設計書.md` | 該当セクションを実装内容と突き合わせて更新 |

機能ループ F7 / 修正ループ R7 の直前に staleness ゲート（`autodev/scripts/check_design_sync.sh`）を実行し、設計書と実装の乖離が無いことを最終確認する。

### 9.9 タスク-設計書紐付けゲート

機能ループ F3・修正ループ R4 のタスク分割完了後、および各イテレーション完了時に `autodev/scripts/check_task_design_link.sh` を実行する。

- 各タスクに `design_doc_ref` が設定されているか確認。
- 関連する設計書の全セクションが少なくとも1つのタスクから参照されていることを確認。
- 機能要件書 / 修正対応書の受け入れ条件がタスク列で網羅されていることを確認。
- いずれかが欠けていれば「分割漏れあり」として F4 / R5 への進行を禁止する。

### 9.10 DB スキーマ管理と DB 切替

- **マイグレーションは Room Migration 一本**。CREATE 文を直に書く・テスト内で create_all する等の経路は禁止する。
- **本番初期化経路**: `python （Room は初回起動時に自動初期化）` が `Room migration（APK 内で実行）` を内部で呼ぶ（実行は `autodev/scripts/run_gradle.sh （Room は初回起動時に自動初期化）` 経由）。
- **テストも本番経路で**: JUnit 起動時の fixture が `init_db.py` を呼ぶ。テスト内で `Base.metadata.create_all()` を呼ぶことを禁止する。
- **SQLite ⇄ （Android 版では非該当: Room 固定）**:
  - 接続文字列を `（Android 版では非該当）DATABASE_URL` 環境変数で受ける。
  - Room の Engine 作成は `app/app/db.py` のみで行う。
  - SQLite 固有機能、MySQL 固有機能を使用しない。
  - 切替検証: `autodev/scripts/check_room_schema.sh` が、ローカルで両方を起動し動作確認。
- **設計書同期**: `docs/設計/data_model.md` は実 DB スキーマから生成。

### 9.11 進捗.md の更新タイミングと内容

- AIエージェントは状態 JSON を書き込んだ直後に **必ず** `autodev/scripts/update_progress_md.sh` を呼ぶ（PostToolUse hook が自動）。
- 同スクリプトは状態 JSON 群を走査して、進捗.md の各セクションを再生成または差分更新する。
- 生成内容は付録D 参照。
- 進捗.md の `last_updated_at` フィールドが状態 JSON の最新 `closed_at` `started_at` 以降になるよう保証する。
- 進捗.md が肥大化しないよう、長大な履歴（30件超）は `autodev/state/history/progress_<日付>.md` に退避する。

### 9.12 ラッパー経由実行の徹底手順

- AIエージェントは以下のコマンドを **直接実行してはならない**。
  - `gradle ...` `./gradlew ...`
  - `adb ...`
  - `emulator ...` `avdmanager ...` `sdkmanager ...`
  - `maestro ...` `apksigner ...`
- 代わりに以下を使う。
  - `autodev/scripts/run_gradle.sh <task...>`（または `build.sh` / `test_unit.sh` / `test_ui.sh`）
  - `autodev/scripts/run_adb.sh <args...>`（または `install.sh` / `run.sh` / `screenshot.sh` / `logcat.sh`）
  - `autodev/scripts/start_emulator.sh` / `stop_emulator.sh` / `setup_android_env.sh`
  - `autodev/scripts/run_maestro.sh ...`（または `test_maestro.sh`）
- 各ラッパーは環境不足を検出した場合、`bash setup.sh`（冪等）の実行を案内する。
- ラッパー経由のコマンド実行ログは `autodev/logs/<日付>/` に追記される。
- PreToolUse hook が素のコマンド呼び出しを検出して fail させる。

### 9.13 実機検証（Compose UI Test / Maestro）の徹底手順

実機検証時の標準手順を以下に固定する。

1. ヘルスチェック実行（`autodev/scripts/healthcheck.sh`）。
2. シナリオ単位の Compose UI Test / Maestro flow ファイル（`maestro/<対象名>.yaml`）を作成・更新。
3. テスト実行（`autodev/scripts/run_maestro.sh test maestro/<対象名>.yaml --reporter=list,html`）。
4. PASS の場合: スクショを `autodev/evidence/screenshots/<対象名>/` に保存、トレースは `autodev/evidence/maestro デバッグ出力/<対象名>/` に保存。
5. FAIL の場合: fail 内容（エラーメッセージ、失敗時スクショ、トレース）をイテレーションループに戻す。
6. attestation に証跡パスを記録。
7. `check_real_device_evidence.sh` で証跡有無を最終確認。
8. 進捗.md を再生成。

ヘッドフル目視確認を行う場合: `HEADFUL=1 autodev/scripts/run_maestro.sh test ...` で実行。AIエージェント自身が勝手にヘッドフル既定にすることは禁止。

### 9.14 `/compact` 実行の標準手順 【廃止】

**本節は廃止した（FR-5.17 廃止に伴う）。** `/compact` はユーザー入力専用でエージェント／フックからは発行
できないため標準手順は無い。コンテキスト逼迫時は Claude Code 組み込みの自動 compact が安全網として作動する。

### 10.3 ループ実装の擬似コード

擬似コード全文は付録C。要点は以下。

- ループは AIエージェント自身の手続きとして実装される（外部スケジューラを置かない）。
- AIエージェントは CLAUDE 規約、`.claude/skills/`、`autodev/scripts/` を頼りに、各ステップを順次実行する。
- 状態の永続化は `autodev/state/` への JSON 書き出しで行う。AI のセッションメモリだけに頼らない。
- 各ステップで必要なシェルスクリプトを呼ぶ。
- Claude Code の hooks がライフサイクル各点で自動補助する。

### 10.4 状態ファイル

詳細スキーマは付録B。最小要件は以下。

- 1ファイルあたり数 KB 〜 数十 KB を上限の目安とする。
- 整形済み JSON / YAML で人間が diff で読める。
- atomic 更新（一時ファイル → rename）。
- Git tracked。
- 監査可能な変更履歴が git log で辿れる。

### 10.5 ハーネス/設定の棚卸し

- `autodev/HARNESS_NOTES.md` に、ハーネスの主要設定とその採用理由を記述する。
- 棚卸し周期: Claude Code 本体のメジャー更新後、および 3〜6ヶ月毎。
- 棚卸し項目: 環境ゲート方針、検証層、同期設計書運用、除外設定、各種規約、hooks 仕様、skills 一覧、`check_release_clean.sh` 対象、ペルソナ駆動テスト頻度、ラッパースクリプトの引数仕様。
- 旧モデル時代の回避策が陳腐化していないか確認し、不要化したものは撤去する。
- DRI の名前を `HARNESS_NOTES.md` 冒頭に記載する。
- `AUTODEV_DISABLE_HOOKS=1` エスケープハッチの使用履歴を必ず記録する。

### 10.6 hook / ゲートによる自動強制

- **Git pre-commit hook** (`autodev/hooks/pre-commit`): `build/` `local.properties` keystore 類 / `autodev/secrets/` / DB・APK ファイルのコミット禁止 / 進捗.md 同期チェック軽量版。
- **Git pre-push hook** (`autodev/hooks/pre-push`): 機能・修正ループのクローズ条件を機械確認（実機検証証跡存在、staleness PASS、デッド資産 PASS、タスク-設計書紐付け PASS、進捗.md 同期 PASS、`.claude/skills/` と `autodev/skills/` 同期 PASS、`check_release_clean.sh` PASS）。
- **Claude Code hooks**: 第10章 10.11 / 付録G 参照。

### 10.7 開発記録・コミット規約

- **コミットメッセージ**: 1行目に `<種別>: <要約> [iter:<対象名>:<番号>]`、本文に変更概要と影響範囲。
- **種別**: `feat`, `fix`, `refactor`, `docs`, `test`, `chore`, `verify`（実機検証コミット）, `persona`（ペルソナテスト由来の修正）。
- **コミット粒度**: イテレーションごとに1コミット以上。1コミットに複数イテレーションをまとめない。
- **プッシュ**: 機能・修正ループのクローズ後にまとめてプッシュ。force push 禁止。
- **コミット対象の制限**: `autodev/db/`, `autodev/logs/`, `autodev/evidence/` 配下はコミットしない。

### 10.8 進捗.md の再生成ロジック

`autodev/scripts/update_progress_md.sh` の責務:

1. `autodev/state/loops/system.json` を読み、システムループの進捗概要を取得。
2. `autodev/state/loops/<対象名>.json` を全走査し、各機能・修正ループの状態を取得。
3. `autodev/state/tasks/<対象名>.json` を読み、現在進行中のループのタスクと進捗を取得。
4. `autodev/state/iterations/<対象名>/` を走査し、直近のイテレーション履歴（10件）を取得。
5. `autodev/state/personas/personas.json` と `autodev/state/persona_tests/` を読み、ペルソナテスト履歴を取得。
6. `autodev/state/attestations/` を読み、最近完了した attestation を取得する。
7. `autodev/logs/<日付>/` の最終行を読み、直近のエラー有無を取得。
8. 以上を所定のテンプレート（付録D）に流し込み、`autodev/開発進捗状況.md` を atomic に書き出す（一時ファイル → rename）。
9. 旧バージョンが大きくなりすぎている場合（500 行超）は古い履歴を `autodev/state/history/progress_<日付>.md` に退避してから新版を書き出す。

スクリプトは bash + jq（+ 補助的に system python3）で実装することを推奨する。

### 10.9 ラッパースクリプト群の実装規約

- 各ラッパースクリプトは bash で実装し、`set -euo pipefail` を冒頭で宣言する。
- 引数はそのまま下流コマンドに pass-through する。
- 標準出力・標準エラーをログファイルに tee する。
- 環境変数 `AUTODEV_NO_WRAPPER=1` が設定されている場合のみ、ラッパー検査を回避できる（緊急時のエスケープハッチ）。

### 10.10 リリース手順

詳細は付録I。要点は以下。

```
1. autodev/scripts/check_release_clean.sh を実行し PASS を確認
2. rm -rf autodev/
3. rm -rf .claude/
4. rm -f CLAUDE.md
5. rm -f */CLAUDE.md  （必要に応じて）
6. git checkout -b release-<バージョン>
7. git add -A && git commit -m "chore: prepare release <バージョン>"
8. 顧客に引き渡し
```

### 10.11 Claude Code hooks の実装規約

- 各 hook スクリプト（`autodev/hooks/claude/*.sh`）は bash で実装し、`set -euo pipefail` を冒頭で宣言する。
- 標準入力で Claude Code から渡される JSON コンテキストを `jq` で解析し、必要なフィールド（tool name、tool args、session info）を取り出す。
- 標準出力は AI への通知メッセージ、終了コード 0 は正常通過、非 0 はツール実行を fail させる（PreToolUse など fail させる hook で使用）。
- 各 hook の動作内容を `.claude/settings.json` の matcher と組み合わせて、副作用が最小になるよう範囲を絞る。
- 各 hook のテストを `autodev/hooks/claude/tests/` 配下に配置（bash + bats など）。
- hook 自体のバグでループが止まることがないよう、hook の冪等性と失敗時の挙動を慎重に設計する。重要 hook 以外は失敗を warn に留める。
- **hook ファイル命名規約**: Claude Code の hook イベント名（PostToolUse / SessionStart など PascalCase）を snake_case に変換し、先頭に `on_` を付ける。
  - `SessionStart` → `on_session_start.sh`
  - `PreToolUse` (matcher: Bash) → `on_pre_tool_use_bash.sh`
  - `PostToolUse` (matcher: Edit|Write) → `on_post_tool_use_edit_write.sh`
  - `PostToolUse` (matcher: Bash + git commit) → `on_post_tool_use_git_commit.sh`
- 各 hook スクリプトは `cd "$CLAUDE_PROJECT_DIR"` を冒頭で行う（または絶対パスで自リポジトリ内のファイルを参照する）。Claude Code から渡される実行時 cwd は信頼しない。

### 10.12 Claude Code skills の実装規約

- 各 SKILL.md は YAML front-matter（`---` で囲んだ **`name` と `description` のみ** が Anthropic Skills の標準フィールド）と本文 Markdown で構成。AI は `description` の内容から発動可否を判断するため、いつ・何のために使う skill かを description に簡潔に記述する。`trigger` や `version` は標準フィールドではないので、必要なら HTML コメントとして残す。
- 本文は階層的に薄く、詳細は autodev/scripts / docs/設計 / 状態ファイル へのリンクで委ねる。
- skill 内で別 skill を呼ぶ場合は `[autodev-gradle-env](../autodev-gradle-env/SKILL.md)` のようにリンクで明示。
- `autodev/scripts/sync_claude_dir.sh` が `autodev/skills/` の変更を `.claude/skills/` にシンボリックリンクまたはコピーで反映する。pre-push hook にも組み込み、同期状態を担保。Windows ネイティブで運用する場合は `--copy` オプションでコピー方式に切り替える。
- 各 SKILL.md には版数欄を設け（HTML コメント可）、変更履歴を簡潔に記載する。

## 第12章 制約事項

本章は本ツールが従わなければならない制約を明示する。違反は受け入れ条件の不合格を意味する。

- **C-12.1**: AIエージェントの起動は `claude --dangerously-skip-permissions` のみ。`claude -p` 等の追加料金が発生する起動形態を使ってはならない。
- **C-12.2**: 並列開発禁止。複数機能・複数イテレーション・複数修正を並列実行してはならない。サブエージェント機能を使った編集の並列化も禁止。読み取り専用の探索利用のみ範囲で許可。
- **C-12.3**: 単一 dev 環境マスタ。worktree symlink ファーム、複数 SDK/AVD 環境の無秩序な並存、複製環境の構築を行ってはならない。
- **C-12.4**: Kotlin + Jetpack Compose + Room + DataStore（+ 必要時 Retrofit / CameraX）。これ以外のスタックは本ツールの対象外。
- **C-12.5**: 実機検証必須。Robolectric / プレビュー / mockで機能・修正ループのクローズ条件を満たしてはならない。代替する場合は明示フラグ + 理由を記録し、ループは閉じない（人手承認待ち）。
- **C-12.6**: 設計書は実装から同期する下流成果物。設計書を変えてから実装を合わせる流れを取ってはならない。ただし全体設計書および機能設計書の初版は機能開発前に新規作成可能。
- **C-12.7**: 全スイート毎回実行禁止。変更領域に限定したテストのみ自動実行する。網羅テストは別フェーズ。
- **C-12.8**: self-report 不信。AIエージェントの「テスト通りました」をハーネスが ground-truth 再実行で検証する。
- **C-12.9**: 巨大状態ファイル禁止。1ファイル数十 KB 程度を上限の目安とし、肥大化した場合は分割する。
- **C-12.10**: 本番環境上での本ツール使用禁止。`--dangerously-skip-permissions` 前提のため、使い捨て可能な dev 環境での使用に限定する。
- **C-12.11**: AI セッションメモリだけに依存する状態を持ってはならない。すべての状態は `autodev/state/` に永続化する。
- **C-12.12**: 生成物（`build/`, `.gradle/`, `local.properties`, keystore, APK）を VCS に載せてはならない。tracked 化を機械検出する。
- **C-12.13**: 本ツール関連のプログラム・データ・入力ファイル・証跡・ログはすべて `autodev/` 配下と `.claude/` 配下に集約する。アプリ層（app/、maestro/、docs/設計/）から `autodev/` `.claude/` 配下を import / require / read してはならない。`check_release_clean.sh` で機械検出する。
- **C-12.14**: タスク分割時に設計書のセクションと紐付けない（`design_doc_ref` 未設定の）タスクを生成してはならない。`check_task_design_link.sh` で機械検出する。
- **C-12.15**: ペルソナ駆動テストの実施タイミング（3機能ごと + 全機能完了後）をスキップしてはならない。スキップする場合は明示フラグ + 理由を記録し、システムループは閉じない。
- **C-12.16**: 状態 JSON（`autodev/state/` 配下のいずれか）を更新した場合、必ず同一の手順内で `autodev/開発進捗状況.md` を再生成する。片方だけ更新された状態を残してはならない。`check_progress_md_sync.sh` で機械検出する。
- **C-12.17**: gradle / gradlew / adb / emulator / avdmanager / sdkmanager / maestro / apksigner を **直接呼び出してはならない**。必ず `autodev/scripts/` のラッパー経由で実行する。
- **C-12.18**: 実機検証は必ず Maestro / Compose UI Test を使用する。Node の素の HTTP クライアントや Robolectric、フロント単体での fetch で代替してはならない。Maestro / Compose UI Test 実行は `autodev/scripts/run_maestro.sh` または `autodev/scripts/run_real_device_check.sh` 経由とする。
- **C-12.19**: 初期雛形時点で Maestro / Compose UI Test のブラウザバイナリ（`maestro install --with-deps chromium`）がインストール済みでなければならない。
- **C-12.20** 【廃止】: `/compact` 関連の制約は撤去した（FR-5.17 廃止）。`/compact` はユーザー入力専用で自動化不可・組み込み自動 compact が安全網。
- **C-12.21**: ハーネス強制は Claude Code の hooks 機能で実装し、CLAUDE 規約のテキストにのみ依存してはならない。FR-5.15.2 に列挙された必須 hooks の不存在は受け入れ条件の不合格を意味する。
- **C-12.22**: 各ループ・各検証の標準手順は Claude Code の skills 機能で配信し、SKILL.md 不在のままドキュメント文章にのみ書き写すことを禁止する。FR-5.16.2 の必須 skills の不存在は受け入れ条件の不合格を意味する。
- **C-12.23**: `.claude/skills/` と `autodev/skills/` の乖離は pre-push hook で fail とする。常に同期した状態を維持する。

| リスク | 影響度 | 対応方針 |
|---|---|---|

## 第17章 用語集

- **AutoDev for Android**: 本書が定める自動開発ツール全体の呼称（仮称）。
- **AIエージェント**: `claude --dangerously-skip-permissions` で起動された Claude のセッション。
- **dev 環境**: 開発・実機検証を行うローカルの実環境。
- **イテレーション**: 1タスク分の最小開発単位。コミットで閉じる。
- **機能ループ**: 1機能の開発を完結させるループ（F1〜F7）。
- **修正ループ**: 1件の修正対応（要件変更・バグ修正）を完結させるループ（R1〜R7）。
- **システムループ**: システム全体開発フロー（S1〜S15）の総称。
- **クローズ条件**: ループが閉じる（次に進める）ために必要な条件群。可逆。
- **実機検証**: 実機（エミュレータ/実機）/ 実 HTTP / 本番初期化経路の実 DB に対する動作確認。Robolectric / プレビュー / mockは含まれない。
- **本番初期化経路**: `init_db.py` のように、本番運用と同じ手順で DB を構築する経路。
- **attestation**: 完了の証拠付き宣言。誰が何をどの実機検証で確認したかを記述し、証跡へのパスを含む。修正ループの attestation は impact_analysis を含む。
- **staleness ゲート**: 設計書と実装の乖離を機械検出する仕組み。
- **scoped tests**: 変更領域 + 影響範囲 + 全体スモークに限定したテスト。
- **DRI**: 説明責任者（Directly Responsible Individual）。ハーネスの整合に責任を持つ1人。
- **ground-truth 再実行**: 自己申告を信用せず、ハーネス側で同じテストを再実行して実態を確認する仕組み。
- **ペルソナ駆動テスト**: 事前定義された複数ペルソナを基に、AI が各ペルソナの視点で実機シナリオを実行し改善要望を生成するテスト形態。
- **全体設計書**: システムループ S4 で AI が新規作成するシステム全体の機能間関係・データフロー・共通機能設計書。`docs/設計/全体設計書.md`。
- **影響範囲分析**: 修正ループ R2 で実施する、変更が及ぶファイル・テーブル・API・画面・データフローの特定作業。
- **design_doc_ref**: タスクが参照する設計書のセクション（ファイルパス + 見出し or 行範囲）。タスク分割漏れ防止のため必須。
- **要件定義書.md**: システム全体の要件を記述したユーザー作成ファイル。`autodev/inputs/要件定義/要件定義書.md`。
- **機能要件_<機能名>.md**: 機能単位の要件を記述したユーザー作成ファイル。
- **修正_<修正名>.md**: 修正対応の要件を記述したユーザー作成ファイル（または AI 自動生成ファイル）。
- **顧客提供物**: リリース時に顧客に引き渡される成果物。app/、maestro/、gradle一式、docs/設計/、README.md、操作手順書、デプロイ手順を含む。`autodev/` 配下と `.claude/` 配下は含まれない。
- **開発進捗状況.md**: 状態ファイル群の人間可読サマリ。`autodev/開発進捗状況.md`。状態更新と同期して再生成。
- **ラッパースクリプト**: AIエージェントが素のコマンドを直接呼ばないよう必ず経由するシェルスクリプト群。`autodev/scripts/run_*.sh`。
- **Android 開発環境**: JDK 17 / Android SDK / AVD / Gradle wrapper / Maestro の総称。setup.sh がユーザー領域に自動導入し、ラッパーがパスを解決する。
- **Maestro / Compose UI Test トレース**: Maestro / Compose UI Test の `trace: on-first-retry` 等の設定により取得されるテスト実行のステップ録画。
- **hooks**: Claude Code のライフサイクルイベント（PreToolUse / PostToolUse / SessionStart / UserPromptSubmit など）で自動実行されるスクリプト群。`.claude/settings.json` で宣言する。
- **skills**: Claude Code が特定タスクで自動発動する手順書ディレクトリ。`.claude/skills/<skill名>/SKILL.md`。
- **PostToolUse hook**: ツール（Edit/Write/Bash 等）実行直後に発火する hook。本ツールでは進捗.md 同期・ゲート起動に使う。
- **PreToolUse hook**: ツール実行直前に発火する hook。本ツールでは禁止コマンド検出に使う。
- **`$CLAUDE_PROJECT_DIR`**: Claude Code 実行時の環境変数。プロジェクトルートを指す。hook コマンドのパス起点として使う。

```

- `items[].expected_layers[]`: 当該カテゴリで期待される実装 layer（Android 固有 / FR-5.21.2.3）。
- `items[].covered`: スクリプトの機械判定。`design_section` / `task_ids` / `implementation_evidence` の3者がすべて非空、かつ `task_layers` が `expected_layers` のいずれかにマッチすれば true。
- `items[].covered_by_ai`（任意 / AI が追記）: 機械判定で false でも実装の実体を AI が確認した場合に `true` を立てる。**判定理由は `ai_reason` フィールドに必ず記録する**（空文字列禁止 / 言い訳語禁止 / NFR-6.3.1 監査性 + 規約14）。
- `items[].ai_reason`（任意）: `covered_by_ai=true` のときの根拠。grep で false-negative になった原因、実体確認した実装ファイル名・関数名等を記述する。
- `items[].testtag_candidates[]`（任意 / 画面要素カテゴリのみ）: 要件書から抽出した testTag 候補。
- `items[].testtag_hits[]`（任意 / 画面要素カテゴリのみ）: `Modifier.testTag("候補")` が grep ヒットした実装ファイルパス。
- `passed`: スクリプトは `(covered OR covered_by_ai)==false` の件数を `items_with_gap` として数え、ゼロなら true とする。**check_traceability.sh は次回起動時、既存出力 JSON の `covered_by_ai` / `ai_reason` を `(category, item)` 一致で承継する**ため、AI が一度付けた判定は次サイクルでも維持される（FR-5.5.5 ground-truth 補完の踏襲）。
- `items_with_ai_pass`: `covered=false` だが `covered_by_ai=true` で PASS 扱いとなった件数。`HARNESS_NOTES.md` / 進捗.md 監査用。
- F7 / R7 attestation は `traceability_ref` に本ファイルのパス、`traceability_passed` に最終結果を含める。

#!/usr/bin/env bash
set -euo pipefail

INPUT=$(cat)

# 禁止コマンドのパターン
FORBIDDEN_PATTERNS=(
  '^[[:space:]]*python[[:space:]]'
  '^[[:space:]]*python3[[:space:]]'
  '^[[:space:]]*pip[[:space:]]'
  '^[[:space:]]*pip3[[:space:]]'
  '^[[:space:]]*JUnit[[:space:]]'
  '^[[:space:]]*adb[[:space:]]'
  '^[[:space:]]*アプリ起動[[:space:]]'
  '^[[:space:]]*maestro[[:space:]]'
)

for pattern in "${FORBIDDEN_PATTERNS[@]}"; do
  if echo "$COMMAND" | grep -qE "$pattern"; then
    echo "ERROR: 素のコマンド実行は禁止です。autodev/scripts/run_*.sh ラッパー経由で実行してください。"
    echo "  対象コマンド: $COMMAND"
    echo "  例: python ... → autodev/scripts/run_gradle.sh ..."
    exit 2  # ツール実行を fail させる
  fi
done

exit 0
